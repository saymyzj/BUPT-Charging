## 2. 通用规范

### 2.1 基础URL

```
http://<服务器IP>:8080
```

### 2.2 统一响应格式

**成功响应：**
```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

**错误响应：**
```json
{
  "code": 1001,
  "message": "Invalid parameters",
  "data": {}
}
```

### 2.3 错误码定义（已冻结）

| 错误码 | 含义 | HTTP状态码 | 使用场景 |
|--------|------|------------|----------|
| 0 | 成功 | 200 | 正常返回 |
| 1001 | 请求参数错误 | 400 | 字段缺失、类型错误、非法值 |
| 1002 | 请求不存在 | 400 | request_id找不到 |
| 1003 | 当前状态不允许该操作 | 400 | 状态机非法跳转 |
| 1004 | 过号，无法确认到场 | 400 | 确认时已过号 |
| 1005 | 已取消，无法继续操作 | 400 | 对取消状态的操作 |
| 1006 | 充电桩故障 | 400 | 桩故障时操作 |
| 1007 | 支付失败 | 400 | 余额不足等 |
| 1099 | 服务器内部错误 | 500 | 未捕获异常 |

### 2.4 时间格式

所有时间字段使用 **ISO 8601** 格式：
```
2026-03-31T10:00:00
```

### 2.5 枚举值定义

**charge_mode（充电模式）：**
- `FAST` - 快充
- `SLOW` - 慢充

**request_status（请求状态）：**
- `PENDING` - 待处理（初始状态）
- `WAITING` - 等待中（已入池）
- `CALLED` - 已叫号
- `CONFIRMED` - 已确认到场
- `CHARGING` - 充电中
- `COMPLETED` - 已完成
- `COMPLETED_EARLY` - 提前完成
- `INTERRUPTED` - 已中断
- `CANCELLED` - 已取消
- `NO_SHOW` - 过号
- `FAULT_REQUEUE` - 故障重排
- `WAITING_TO_LEAVE` - 等待挪车

**station_status（充电桩状态）：**
- `IDLE` - 空闲
- `RESERVED` - 已预留
- `CHARGING` - 充电中
- `WAITING_TO_LEAVE` - 等待挪车
- `FAULT` - 故障

## 4. 接口详细定义

### 4.1 GET /health

**功能：** 服务健康检查

**请求参数：** 无

**响应示例：**
```json
{
  "code": 0,
  "message": "success",
  "data": {
    "status": "ok",
    "timestamp": "2026-03-31T10:00:00"
  }
}
```

---

### 4.2 POST /api/request/create

**功能：** 提交充电请求

**请求体：**
```json
{
  "request_time": "2026-03-31T10:00:00",  // 必填，ISO 8601格式
  "charge_mode": "FAST",                   // 必填，枚举：FAST/SLOW
  "request_energy": 20.0                   // 必填，正数，单位：kWh
}
```

**成功响应：**
```json
{
  "code": 0,
  "message": "success",
  "data": {
    "request_id": "REQ001",                  // 请求ID，格式：REQ+序号
    "status": "WAITING",                     // 请求状态
    "estimated_wait_seconds": 600,           // 预计等待秒数
    "estimated_start_time": "2026-03-31T10:10:00",  // 预计开始时间
    "estimated_finish_time": "2026-03-31T11:10:00"  // 预计完成时间
  }
}
```

**错误响应：**
- `1001` - 参数错误（字段缺失、非法值）

---

### 4.3 GET /api/request/status/{request_id}

**功能：** 查询请求状态

**路径参数：**
- `request_id` - 请求ID，如：REQ001

**成功响应：**
```json
{
  "code": 0,
  "message": "success",
  "data": {
    "request_id": "REQ001",
    "status": "WAITING",
    "station_id": "FAST_01",                 // 分配的充电桩编号，可能为null
    "estimated_wait_seconds": 300,           // 预计等待秒数
    "last_called_time": "2026-03-31T10:08:00" // 上次叫号时间，可能为null
  }
}
```

**错误响应：**
- `1002` - 请求不存在

---

### 4.4 POST /api/request/cancel_queue

**功能：** 取消排队

**前置条件：** 请求状态为 PENDING/WAITING/CALLED

**请求体：**
```json
{
  "request_id": "REQ001",                  // 必填
  "cancel_time": "2026-03-31T10:05:00"     // 必填，ISO 8601格式
}
```

**成功响应：**
```json
{
  "code": 0,
  "message": "success",
  "data": {
    "request_id": "REQ001",
    "status": "CANCELLED"
  }
}
```

**错误响应：**
- `1001` - 参数错误
- `1002` - 请求不存在
- `1003` - 当前状态不允许取消

---

### 4.5 POST /api/request/confirm_arrival

**功能：** 确认到场

**前置条件：** 请求状态为 CALLED

**请求体：**
```json
{
  "request_id": "REQ001",                  // 必填
  "confirm_time": "2026-03-31T10:09:00"    // 必填，ISO 8601格式
}
```

**成功响应：**
```json
{
  "code": 0,
  "message": "success",
  "data": {
    "request_id": "REQ001",
    "status": "CONFIRMED",
    "station_id": "FAST_01"                  // 分配的充电桩编号
  }
}
```

**错误响应：**
- `1001` - 参数错误
- `1002` - 请求不存在
- `1003` - 请求不在CALLED状态
- `1004` - 已过号（超过confirm_timeout时间）

---

### 4.6 POST /api/request/interrupt_charge

**功能：** 中断充电

**前置条件：** 请求状态为 CHARGING

**请求体：**
```json
{
  "request_id": "REQ001",                  // 必填
  "interrupt_time": "2026-03-31T10:40:00"  // 必填，ISO 8601格式
}
```

**成功响应：**
```json
{
  "code": 0,
  "message": "success",
  "data": {
    "request_id": "REQ001",
    "status": "INTERRUPTED",
    "actual_energy": 12.5,                   // 实际充电电量（kWh）
    "actual_service_seconds": 1800           // 实际服务时长（秒）
  }
}
```

**错误响应：**
- `1001` - 参数错误
- `1002` - 请求不存在
- `1003` - 请求不在CHARGING状态

**实现说明：**
- 实际充电电量根据充电桩功率和充电时长计算
- 释放充电桩，状态变为IDLE
- 触发重调度

---

### 4.7 POST /api/request/confirm_leave

**功能：** 确认挪车

**前置条件：** 请求状态为 WAITING_TO_LEAVE

**请求体：**
```json

  "leave_time": "2026-03-31T11:15:00"      // 必填，ISO 8601格式
}
```

**成功响应：**
```json
{
  "code": 0,
  "message": "success",
  "data": {
    "request_id": "REQ001",
    "status": "COMPLETED"
  }
}
```

**错误响应：**
- `1001` - 参数错误
- `1002` - 请求不存在
- `1003` - 请求不在WAITING_TO_LEAVE状态

**实现说明：**
- 释放充电桩，状态变为IDLE
- 触发重调度

## 5. 状态流转图

```
                    +-----------+
                    |  PENDING  |  (初始状态)
                    +-----+-----+
                          |
                          v
                    +-----------+
         +--------->|  WAITING  |<---------+
         |         +-----+-----+          |
         |               |                |
         |    叫号       v   用户取消      |  故障
         |         +-----------+          |
         +---------|   CALLED  |----------+
                   +-----+-----+
                         |
              确认到场   v   超时过号
                   +-----------+
                   | CONFIRMED |
                   +-----+-----+
                         |
              开始充电   v
                   +-----------+
         +-------->|  CHARGING |<---------+
         |         +-----+-----+          |
         |               |                |
    充电完成             v   用户中断      |
         |         +-----------+          |
         +---------| WAITING_TO|----------+
                   |  _LEAVE   |
                   +-----+-----+
                         |
              确认挪车   v
                   +-----------+
                   | COMPLETED |
                   +-----------+

其他终态：
- CANCELLED (取消)
- INTERRUPTED (中断)
- NO_SHOW (过号)
- FAULT_REQUEUE (故障重排)
```

## 附录A：与原版冻结接口文档的差异说明

本文档基于原《冻结接口文档.md》（以下简称"原文档"）进行实现和更新，以下是主要差异：

### A.1 响应格式统一化

| 项目 | 原文档 | 本文档（已实现） | 差异说明 |
|------|--------|------------------|----------|
| /health 响应 | 直接返回 `{"status": "ok"}` | 统一包装为 `{"code": 0, "message": "success", "data": {"status": "ok"}}` | 所有接口统一使用 `code/message/data` 格式 |

**原因：** 为保证所有接口响应格式一致性，健康检查接口也采用统一响应包装。

### A.2 取消排队前置条件扩展

| 项目 | 原文档 | 本文档（已实现） | 差异说明 |
|------|--------|------------------|----------|
| 可取消状态 | `WAITING` 或 `CALLED` | `PENDING`、`WAITING` 或 `CALLED` | 增加了 `PENDING` 状态 |

**原因：** 在简化实现中，请求创建后初始状态为 `PENDING`，允许用户在此状态取消更符合用户体验。

### A.3 错误码补充HTTP状态码

| 项目 | 原文档 | 本文档（已实现） | 差异说明 |
|------|--------|------------------|----------|
| 错误码定义 | 仅定义code值 | 增加HTTP状态码列（200/400/500） | 明确HTTP响应码 |

**实现映射：**
- `code = 0` → HTTP 200
- `1000 <= code < 2000` → HTTP 400（客户端错误）
- `code >= 2000` → HTTP 500（服务器错误）

### A.4 数据库表字段补充

相比原文档，本文档的数据库表增加了以下字段：

**charge_request 表：**
- `battery_limit_energy` - 电池上限电量（kWh）
- `fault_requeue_flag` - 故障重排标记

**charging_session 表：**
- `leave_deadline` - 挪车截止时间（用于计算占位费）

### A.5 新增实现状态标记

本文档为每个接口增加了实现状态标记：
