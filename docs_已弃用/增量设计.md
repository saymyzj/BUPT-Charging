# 智能充电桩调度计费系统——增量设计

> 项目团队：G7小组  
> 编制日期：2026年4月  
> 文档版本：V5.0  
> 文档定位：在最终验收口径下，基于现有功能设计与老师“至少三次增量”的要求，对系统交付节奏、功能优先级和阶段目标进行重新拆分

---

## 一、设计背景与调整说明

### 1.1 项目目标

本系统面向校区智能充电场景，目标是在模拟环境下完成：

- 充电请求受理
- 动态调度与排队管理
- 充电过程监控
- 计费结算与结果输出
- 管理端监控与验收测试适配

在调度目标上，系统优先追求：

- 缩短单用户总完成时间
- 降低平均等待时间
- 提高充电桩利用率
- 在取消、过号、故障等异常场景下保持稳定性

### 1.2 原两增量方案的调整原因

原有文档将系统划分为两个增量：

- 增量一：中期验收
- 增量二：最终验收

根据老师最新要求，项目过程必须至少体现三次增量交付。因此本次调整采用如下原则：

1. 不改变系统总体功能边界
2. 不推翻现有功能设计、架构设计和冻结接口口径
3. 将原“两增量”拆分为“三次可独立汇报的阶段性交付”
4. 让每次增量都对应清晰的可演示成果，而不只是代码堆积

本次拆分后的总体思路是：

- 增量一：先做核心主流程闭环
- 增量二：补齐异常处理与验收接口闭环
- 增量三：完成参数化验收适配与管理端完善

### 1.3 总体设计原则

系统仍采用以下核心方案，不因增量拆分而改变：

`中央等待池 + 动态调度 + 临近确认 + 事件驱动重排`

原因如下：

- 固定绑定某个桩队列，难以处理取消、过号、故障和提前充满
- 中央等待池更适合统一计算预计等待时间
- 动态分配最终桩号，更符合实际调度系统运行方式
- 最终验收要求灵活输入场景参数，该方案更易做兼容适配

### 1.4 最终验收口径下的设计约束

根据现有正式文档，增量设计必须满足以下约束：

- 充电桩数量、等待区容量等不是固定值，而是场景参数
- `waiting_area_capacity` 是全局共享等待区容量
- “充电桩排队队列长度”需要兼容两种解释：
  1. `UNIFORM_CAPACITY`
  2. `STATION_SNAPSHOT`
- 运行时唯一的提交容量约束是 `waiting_area_capacity`
- `station_queue_capacity` 与 `station_snapshots.queue_length` 仅用于验收输入兼容和初始化转换
- 单次请求、批量模拟、详单、账单和评测指标都必须保持统一口径
- 最终验收重点是系统本身能否正确建模、调度、计费和输出结果

## 二、增量划分的前提确认

### 2.1 角色与核心需求

| 角色 | 描述 | 核心需求 |
|------|------|----------|
| 普通用户 | 校园内发起充电请求的用户 | 提交请求、查看等待、接收叫号、查看详单账单 |
| 管理员 | 负责系统运行和资源管理的人员 | 监控桩状态、处理异常、配置参数、查看统计、管理设备 |

### 2.2 用户提供的数据

用户发起充电申请时，最小输入保持不变：

| 字段 | 含义 |
|------|------|
| `charge_mode` | 充电类型，取值为快充或慢充 |
| `request_energy` | 申请充电量，作为调度预测输入 |

系统据此计算：

- 预计等待时长
- 预计开始时间
- 预计完成时间
- 预计服务时长

补充说明：

- `request_energy` 只作为预测输入，不直接等于最终结算量
- 系统在充电过程中记录 `actual_energy`
- 若车辆提前充满，则提前结束并触发重调度
- 最终费用按 `actual_energy` 和实际服务时长计算

### 2.3 核心业务主线

系统完整主线保持不变：

1. 用户提交充电请求
2. 请求进入对应类型中央等待池
3. 调度器计算预计等待时间与预计完成时间
4. 某桩临近可用时触发叫号
5. 用户确认到场
6. 由服务端内部事件同步推进开始充电并记录会话
7. 充电完成或提前充满
8. 用户确认离场
9. 系统生成详单与账单
10. 若发生取消、过号、中断、故障等事件，则触发重调度

补充说明：

- `NO_SHOW` 原请求保留终态；若允许继续排队，应生成新的请求重新入池
- 故障续充采用“原请求保留记录 + 新请求重新入池”的方式
- “开始充电”不新增公开接口

### 2.4 系统边界

- 充电桩采用模拟设备，不接入真实硬件
- 支付采用模拟支付
- 用户是否到场、是否及时挪车等行为由系统状态模拟
- 项目以课程实验和最终验收为目标，不追求生产级吞吐与高可用

## 三、三次增量设计

### 3.1 划分原则

三次增量划分遵循以下原则：

1. 增量一必须形成最小可演示闭环
2. 增量二必须补齐核心异常分支和验收接口能力
3. 增量三必须完成参数化、统计分析和最终验收适配
4. 每次增量都要有明确的输入、处理、输出和展示成果
5. 原功能设计中的总体优先级不变，只调整交付顺序

### 3.2 增量总览

| 增量 | 交付定位 | 目标描述 | 主要成果 |
|------|----------|----------|----------|
| 增量一 | 核心主流程闭环 | 先跑通“提交请求-排队-叫号-充电-结算”的最小闭环 | 动态预测、状态流转、基础计费、基础详单账单 |
| 增量二 | 异常处理与验收闭环 | 在主流程闭环基础上补齐异常分支、批量模拟和统一输出 | 取消/中断/过号处理、批量模拟、统一详单账单指标输出 |
| 增量三 | 参数化与最终验收完善 | 围绕最终验收完成场景参数化配置、管理端完善和展示优化 | 场景参数适配、统计报表、设备与用户管理、支付与导出 |

## 四、功能优先级的总体判断

### 4.1 P0 功能判断标准

以下能力属于必须优先完成的 P0：

- 能形成主流程闭环的能力
- 能影响调度正确性的能力
- 能影响正式验收输入输出口径的能力
- 能影响结果展示和批量测试的能力

### 4.2 P1 功能判断标准

以下能力可作为 P1：

- 不影响主流程正确性的增强页面
- 不影响正式输入输出结构的展示优化
- 对完整度有帮助但不决定验收成败的管理端细化能力

### 4.3 当前优先级结论

本项目当前最重要的三层优先级应理解为：

1. 先保主流程正确
2. 再保验收输入输出稳定
3. 最后做参数化、统计和展示增强

## 五、系统结构与增量实施关系

### 5.1 模块划分

| 模块 | 主要职责 |
|------|----------|
| 用户服务模块 | 注册、登录、身份识别 |
| 请求受理模块 | 接收充电请求，生成调度任务 |
| 等待池管理模块 | 维护快充池和慢充池中的请求 |
| 调度引擎模块 | 计算优先级、预计等待时间和最终分配 |
| 充电桩状态模块 | 维护充电桩状态和最早可用时间 |
| 充电会话模块 | 维护充电中、完成、挪车等状态 |
| 计费与账单模块 | 根据会话记录生成费用与账单 |
| 通知模块 | 叫号、提醒、异常通知 |
| 管理后台模块 | 提供监控、配置、统计和审计能力 |

### 5.2 模块与增量的对应关系

| 增量 | 重点模块 |
|------|----------|
| 增量一 | 请求受理、等待池管理、调度引擎、充电会话、基础计费 |
| 增量二 | 异常处理、批量模拟、统一输出、审计与跨设备访问 |
| 增量三 | 参数配置、统计分析、设备管理、用户管理、报表导出 |

### 5.3 算法与场景适配原则

系统算法主线保持不变：

- 快充请求进入快充等待池
- 慢充请求进入慢充等待池
- 调度器按事件驱动统一重算预计等待时间
- 当桩接近可用时触发叫号和最终分配

为兼容最终验收输入，还必须补充一层场景适配：

- 若输入是统一队列容量，则解释为每桩排队容量上限
- 若输入是桩级状态快照，则解释为初始占用与排队情况
- 两种输入都先转换为内部等待池状态与桩状态，再进入统一调度流程

## 六、当前结论

1. 老师要求“至少三次增量”后，原两增量方案已不再合适
2. 最合理的调整方式不是新增大量功能，而是把原方案拆成“三次可独立汇报的交付”
3. 三次增量应分别对应：
   - 核心主流程闭环
   - 异常处理与验收闭环
   - 参数化与最终验收完善
4. 这样拆分后，功能设计、优先级判断和后续小组分工都更容易对齐
