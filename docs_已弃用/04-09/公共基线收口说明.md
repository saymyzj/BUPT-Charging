# 公共基线收口说明

> 日期：2026-04-09  
> 面向对象：成员B、成员C  
> 目的：说明当前 `develop` 分支已完成的公共收口内容，以及后续协作时必须遵守的公共基线

## 一、本轮收口目标

本轮修改的目标不是继续堆局部功能，而是先把 `develop` 打造成：

1. 三人都可以继续接入的公共分支
2. 搭载成员A当前最新“可扩展、接口一致”的调度内核
3. 尽量减少历史兼容补丁和临时口径
4. 避免成员B、成员C后续继续围绕公共接口返工

当前原则可以概括为：

**内部逻辑允许继续增强，公共接口和公共数据结构必须稳定。**

## 二、本轮代码侧已完成的收口

### 2.1 调度内核公共边界已固定

当前继续作为公共稳定入口的函数为：

1. `predict_request_by_id`
2. `reschedule_waiting_requests`
3. `trigger_reschedule_for_event`
4. `simulate_batch_case`
5. `advance_request_state_for_status_view`

含义：

- 成员B继续做服务端流程时，应通过这些入口接入调度能力
- 成员C继续做页面对接时，不应再假设调度模块内部结构
- 成员A后续继续增强算法时，应优先保持这些入口不变

### 2.2 冻结结构统一由构造器输出

新增：

- `backend/app/services/contract_builders.py`

当前统一由该模块生成：

1. 状态查询响应
2. 详单 `detail`
3. 账单 `bill`
4. 批量模拟 `summary`

意义：

- 后续不再允许在不同模块里手写不同形状的公共返回结构
- 成员B 和 成员C 应将这里视为稳定协议层

### 2.3 主流程状态机做了公共收口

当前 `develop` 已支持以下主流程推进：

1. `WAITING -> CALLED`
2. `CALLED -> CONFIRMED`
3. `CONFIRMED -> CHARGING`
4. `CHARGING -> COMPLETED`
5. `CHARGING -> INTERRUPTED`
6. `COMPLETED / INTERRUPTED -> WAITING_TO_LEAVE`
7. `WAITING_TO_LEAVE -> LEAVE`
8. `CALLED -> NO_SHOW`

补充说明：

- `confirm_arrival` 的对外响应语义已恢复为冻结文档中的 `CONFIRMED`
- 充电开始改为由统一状态推进逻辑处理，不再对外直接在确认到场接口里返回 `CHARGING`
- `NO_SHOW` 原请求保留终态；若允许继续排队，应生成新请求重新入池
- 故障续充采用“原请求保留记录 + 新请求重新入池 + `fault_requeue_flag`”方案
- 运行时唯一的提交容量约束是共享等待区 `waiting_area_capacity`

### 2.4 批量模拟输出已对齐冻结文档

当前 `POST /api/batch/simulate` 已收口为：

1. `summary` 只保留冻结文档要求字段
2. `results[*].detail` 使用冻结的 18 字段结构
3. `results[*].bill` 使用冻结的 9 字段结构

说明：

- 后续成员B继续增强模拟逻辑时，不应再修改这三部分的字段外形

### 2.5 公共测试基线已修复

本轮修复了两个会直接影响公共分支稳定性的底层问题：

1. `init_db` 现在兼容两种调用方式：
   - 传 Flask app
   - 传数据库路径字符串
2. 旧的 `scenario_adapter` 测试链路重新恢复可运行

当前已验证通过：

1. `backend/tests/test_scenario_adapter.py`
2. `backend/tests/test_frozen_contracts.py`
3. `backend/tests/test_auth_minimal.py`

### 2.6 最小鉴权链路已补齐

当前 `develop` 已新增最小可用认证接口：

1. `POST /api/auth/register`
2. `POST /api/auth/login`
3. `GET /api/auth/profile`

实现原则：

1. 用户名唯一校验
2. 基础密码摘要
3. 简单 token 认证

说明：

- 当前不追求复杂权限体系
- 认证链路的目标是“可用、稳定、够联调”，不是“复杂安全方案”
- 不要把后续主要时间消耗在鉴权花样上

### 2.7 登录页已改为真实最小后端调用

当前前端登录页不再只是演示跳转：

1. 注册调用 `/api/auth/register`
2. 登录调用 `/api/auth/login`
3. 成功后保存 token 和 role
4. 根据 role 跳转用户端或管理端

说明：

- 这只是最小公共基线，不代表后续账户体系已经做完
- 但至少 `develop` 现在不再是“登录页有、后端完全没有”

1. 以当前 `develop` 为唯一公共后端基线
