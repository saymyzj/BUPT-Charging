# 智能充电桩调度计费系统：增量设计

> 版本：V3.0
> 状态：最终冻结
> 冻结日期：2026-04-19
> 维护人：组长A
> 目标：在保持当前协作模式不被打散的前提下，重映射增量目标

## 一、增量划分原则

本次需求调整属于大改，但项目已开工，现阶段不推翻原有三增量协作框架，只重排每个增量的交付内容。

增量原则：

1. 先打通最终要求中的严格主流程
2. 再补特殊场景与故障处理
3. 最后完成扩展调度、统计和部署上线收口

## 二、增量一：严格主流程闭环

### 目标

建立完全符合最终要求的普通模式核心流程。

### 必做内容

- 等候区与充电区模型
- 快慢充固定桩数量参数化（F-J01）
- 每桩固定长度队列模型
- `F/T` 排队号生成与重置规则
- 同模式等候区头车优先
- 同模式桩队列最短完成时长分配
- 严格计费口径：峰平谷 + 服务费
- 用户端主流程：注册、登录、创建请求、查看排队号、查看前车数量、查看详单
- 管理端基础查看：桩状态、桩队列、基础报表

### 阶段验收标准

- 普通模式全链路可回放
- 新文档与新代码均不再依赖中央等待池口径

## 三、增量二：特殊场景与故障闭环

### 目标

补齐用户修改请求、取消、提前结束和故障处理。

### 必做内容

- 等候区修改模式（F-U04）
- 等候区修改电量（F-U05）
- 等候区取消请求（F-U06）
- 充电区取消 = 提前结束并结算（F-U07，与 F-U06 合并验证终态 `COMPLETED_EARLY`）
- 故障策略 `PRIORITY`（F-A05）
- 故障策略 `TIME_ORDER`（F-A05）
- 故障标记与恢复、故障后重调度（F-A06）
- 管理员启动/关闭充电桩（F-A01）
- 管理员运行时切换故障调度策略（F-A05）

### 阶段验收标准

- 修改、取消、故障和恢复都能严格按文档回放
- 故障中断产生独立详单，后续调度规则清晰且一致

## 四、增量三：扩展调度、统计与上线收口

### 目标

完成扩展调度能力、报表完善和正式部署上线准备。

### 必做内容

- `EXT_SINGLE_BATCH`（F-A04）
- `EXT_FULL_BATCH`（F-A04）
- 管理员运行时切换普通/扩展模式（F-A04）
- 用户/车辆信息维护接口（F-A07，新增）
- 日/周/月报表完整字段（F-A08）
- 详单 / 账单 / 报表字段字典自动化校验脚本（对齐 [冻结接口文档.md](冻结接口文档.md) §9/§10/§11）
- 系统验收输入输出结构收口（F-J02 / F-J03 / F-J04）
- 单台 ECS + Nginx + Flask + SQLite + HTTPS 部署文档收口（见 [部署上线手册.md](部署上线手册.md)）
- 组长上线职责、备份与回滚预案收口（见 [部署上线手册.md](部署上线手册.md)）

### 阶段验收标准

- 扩展模式可独立说明、独立启用、独立验收
- 上线部署口径写入正式文档，代码后续具备按此方案部署的兼容目标

## 六、当前开发节奏约束

为避免破坏已形成的协作模式，本次重构要求：

- 保留三增量框架
- 不按“前端/后端分层”重新拆团队工作
- 不把所有必做需求都强行压进单一增量
- 先用新文档冻结口径，再推进代码重构
