# 智能充电桩调度计费系统：故障与扩展场景指南

> 版本：V3.0
> 状态：最终冻结
> 冻结日期：2026-04-19
> 维护人：组长A
> 目标：面向答辩/验收/演示的场景剧本集合，与 [调度算法.md](调度算法.md) 配套使用

本指南不定义新规则。规则以 [调度算法.md](调度算法.md) §三/§四为唯一来源，[功能设计.md](功能设计.md) §5/§7 给出交互语义。本指南只给可回放的具体剧本。

## 一、场景编号规则

- `F-CASE-xx`：故障类场景（含故障发生与故障恢复）
- `E-CASE-xx`：扩展调度场景（`EXT_SINGLE_BATCH` 与 `EXT_FULL_BATCH`）

每个场景包含：

1. 参数快照（`fast_station_count / slow_station_count / waiting_area_capacity / charging_queue_len / dispatch_mode / fault_dispatch_mode`）
2. 初始车辆输入序列（`queue_number / charge_mode / request_energy / request_time`）
3. 预期调度结果（桩队列终态 + 每辆车的 `target_station_code / target_queue_position`）
4. 预期详单/账单字段值（仅对已产生详单的车辆，按 [冻结接口文档.md](冻结接口文档.md) §9 字典写明关键字段）

## 二、公共参数快照（场景默认）

除非单场景覆盖，均采用：

```
fast_station_count = 3         慢/快充桩：FAST_01..FAST_03, SLOW_01..SLOW_02
slow_station_count = 2         快充功率 30 kW，慢充功率 10 kW
waiting_area_capacity = 6
charging_queue_len = 2         含充电位
dispatch_mode = NORMAL
fault_dispatch_mode = TIME_ORDER
```

## 三、故障场景

### 3.1 `F-CASE-01`（PRIORITY × 队首故障）

- 故障策略：`PRIORITY`
- 初始车辆：`F1~F6`，均为快充，请求 30/30/30/30/30/30 kWh
- 初始分配：`FAST_01=[F1,F2]`、`FAST_02=[F3,F4]`、`FAST_03=[F5,F6]`
- 触发：`FAST_01` 故障；`F1` 已充 10 kWh
- 预期：
  - `F1` 生成 `FAULT_INTERRUPTED` 详单，`actual_energy=10`、`stop_time=故障时刻`
  - `F1` 以新请求 20 kWh 重入等候区（新排队号）
  - 故障队列 `=[F2]`
  - 当 `FAST_02` 或 `FAST_03` 出现空位时优先补 `F2`
  - 等候区叫号暂停，直到 `F2` 入桩

### 3.2 `F-CASE-02`（PRIORITY × 队尾车辆继续等待）

- 策略同上
- 初始：`FAST_01=[F1,F2]`、`FAST_02=[F3]`、`FAST_03=[F5]`，等候区 `F6,F7`
- 触发：`FAST_01` 故障；`F1` 已充 5 kWh
- 预期：
  - `F1` 生成 `FAULT_INTERRUPTED` 详单后以新请求重入等候区
  - 故障队列 `=[F2]`，`FAST_02 / FAST_03` 空位出现时先补 `F2`
  - 故障队列清空前等候区 `F6, F7` 继续等候
  - 故障队列清空后，恢复叫号，按 `F6, F7` 顺序取车

### 3.3 `F-CASE-03`（PRIORITY × 慢充）

- 同上但模式为慢充，两个慢充桩 `SLOW_01 / SLOW_02`
- 初始：`SLOW_01=[T1,T2]`、`SLOW_02=[T3,T4]`
- 触发：`SLOW_01` 故障；`T1` 已充 2 kWh
- 预期：
  - `T1` `FAULT_INTERRUPTED`，重入等候区
  - 故障队列 `=[T2]`，在 `SLOW_02` 出空位时进入

### 3.4 `F-CASE-04`（TIME_ORDER × 统一重排）

- 故障策略：`TIME_ORDER`
- 初始：`FAST_01=[F1,F2]`、`FAST_02=[F3,F4]`、`FAST_03=[F5,F6]`，均为快充
- 触发：`FAST_01` 故障；`F1` 已充 8 kWh
- 预期：
  - `F1` `FAULT_INTERRUPTED` 后重入等候区
  - 参与重排集合 `= {F2, F4, F6}`（均为未开始充电）
  - 按排队号统一排序：`F2 < F4 < F6`
  - 重新装配到 `FAST_02 / FAST_03`（具体位置按完成时间最短原则，不做跨模式）
  - 重排完成前等候区叫号暂停

### 3.5 `F-CASE-05`（TIME_ORDER × 仅故障桩单车）

- 策略：`TIME_ORDER`
- 初始：`FAST_01=[F1]`、`FAST_02=[F2,F3]`、`FAST_03=[F4]`
- 触发：`FAST_01` 故障；`F1` 已充 0 kWh（尚未开始）
- 预期：
  - `F1` 不生成 `FAULT_INTERRUPTED`（其未开始充电，直接回到重排集合）
  - 参与重排集合 `= {F1, F3}`（`F2` 已在充电不参与）
  - 按 `F1 < F3` 装配到 `FAST_02 / FAST_03`

### 3.6 `F-CASE-06`（TIME_ORDER × 边界）

- 策略：`TIME_ORDER`
- 初始：`FAST_01=[F1,F2]`、`FAST_02=[F3]`、`FAST_03=[F4]`
- 触发：`FAST_01` 故障；`F1` 正在充电、已充 12 kWh
- 预期：
  - `F1` `FAULT_INTERRUPTED`
  - 重排集合 `= {F2}`（`F3 / F4` 均已在充电）
  - `F2` 重新装配到 `FAST_02` 或 `FAST_03` 中完成时间最短的那个

### 3.7 `F-CASE-07`（故障恢复 × 时间顺序重排）

- 前置：执行完 `F-CASE-04`
- 触发：`FAST_01` 恢复
- 预期：
  - 其它同类型桩中仍存在未开始充电车辆 `{F4', F6'}`（经重排后的状态）
  - 暂停等候区叫号
  - 将 `{F4', F6'}` 与 `FAST_01` 空位合并，按排队号重新调度
  - 调度完成后恢复叫号

### 3.8 `F-CASE-08`（故障恢复 × 其它桩已全部进入充电）

- 前置：执行完 `F-CASE-04`，且 `F4 / F6` 已全部进入 `CHARGING`
- 触发：`FAST_01` 恢复
- 预期：
  - 其它同类型桩中无未开始充电车辆
  - 不触发重排
  - `FAST_01` 直接从等候区按 `NORMAL` 规则补位

## 四、扩展场景

### 4.1 `E-CASE-01`（`EXT_SINGLE_BATCH` × 2 空位）

- 场景：`dispatch_mode=EXT_SINGLE_BATCH`
- 初始：`FAST_01 / FAST_02` 同时空一个位，等候区 `F1, F2`（均快充，20 / 10 kWh）
- 预期：
  - 不再遵守等候区头车优先
  - 在 `{F1 → FAST_01, F2 → FAST_02}` 与 `{F2 → FAST_01, F1 → FAST_02}` 之间
    选择 `Σ(wait_time + self_charge_time)` 最小的组合

### 4.2 `E-CASE-02`（`EXT_SINGLE_BATCH` × 模式隔离）

- 初始：`FAST_01 / SLOW_01` 各空一个位，等候区 `F1, T1, F2`
- 预期：
  - 本批候选只能在同模式内挑选：`FAST_01 ← {F1, F2}`，`SLOW_01 ← {T1}`
  - 快慢不可跨配

### 4.3 `E-CASE-03`（`EXT_SINGLE_BATCH` × 多空位）

- 初始：`FAST_01 / FAST_02 / FAST_03` 各空一个位，等候区 `F1, F2, F3`
- 预期：
  - 枚举 3! 种指派，取目标函数最小的组合
  - 无 `PRIORITY`、无 `TIME_ORDER` 语义介入

### 4.4 `E-CASE-04`（`EXT_FULL_BATCH` × 全站刚好满员）

- 场景：`dispatch_mode=EXT_FULL_BATCH`
- 全站总车位 `= 充电区(3×2 + 2×2) + 等候区(6) = 16`
- 初始：恰好 16 辆车到达，混合快/慢请求
- 预期：
  - 本批不区分快慢充
  - 任一车辆可分配到任一桩
  - 目标 `min Σ (wait_time + self_charge_time)`

### 4.5 `E-CASE-05`（`EXT_FULL_BATCH` × 小配比场景）

- 场景同上，但车辆全为慢充请求
- 预期：
  - 可把慢充车分配到快充桩（提高功率缩短完成时间）
  - 不关心模式，但仍遵守桩队列容量约束

### 4.6 `E-CASE-06`（`EXT_FULL_BATCH` × 下一批触发）

- 前置：执行完 `E-CASE-04`，全部车辆完成
- 触发：再来 16 辆车
- 预期：
  - 视为新一批 `EXT_FULL_BATCH`
  - 与上一批互不影响

## 五、互斥边界（冗余提示）

- `EXT_SINGLE_BATCH / EXT_FULL_BATCH` 期间不考虑请求修改与故障
- 任意扩展模式与 `NORMAL` 互斥：切换必须通过 `PUT /api/admin/system/dispatch-mode`
- 故障恢复不分 `PRIORITY / TIME_ORDER`，统一按时间顺序重排
