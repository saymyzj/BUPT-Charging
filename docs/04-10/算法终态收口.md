# 算法终态收口

> 日期：2026-04-10  
> 负责人：成员A  
> 复杂度：large  
> 目标：先统一正式文档口径，再将调度算法内核、异常策略和批量模拟落实到最终态，同时保持公共接口与数据结构不变

---

## 一、问题与背景

当前 `develop` 已完成一轮公共基线收口，公共接口和最小结构已经稳定，但调度算法和异常策略仍存在两类问题：

1. 正式文档之间对部分关键状态和异常分支仍存在冲突或歧义
2. 当前代码实现仍偏“最小可用”，离最终态算法内核还有差距

当前最典型的冲突点包括：

1. `NO_SHOW` 是否原请求终止、是否自动重入队
2. `FAULT_REQUEUE` 是原请求状态还是新续充请求的标记
3. 中断充电后桩是否统一进入 `WAITING_TO_LEAVE`
4. “开始充电”事件是否继续通过内部同步机制处理
5. `station_queue_capacity` 到底是运行时容量约束，还是仅作为验收输入兼容参数

本轮已由组长A拍板：

1. `NO_SHOW` 原请求保留终态；若允许继续排队，则新建请求重新入队
2. 故障续充采用“新建请求 + `fault_requeue_flag`”方案
3. 中断充电后统一进入 `WAITING_TO_LEAVE`
4. 不新增公开“开始充电”接口，只做内部事件同步
5. 共享等待区容量是运行时唯一提交容量约束；桩队列容量仅作为输入兼容和初始化依据

---

## 二、目标

1. 统一文档口径，消除正式文档间的歧义
2. 将上述 5 个拍板点落实到最终态算法与状态机实现
3. 保持当前公共接口、公共字段和调度入口函数不变
4. 让成员B、成员C后续继续接入时不需要返工公共边界
5. 保持 `develop` 作为唯一可共享的公共基线

---

## 三、非目标

1. 不在本轮引入新的公开 HTTP 接口
2. 不把登录注册扩展为复杂权限体系
3. 不在本轮追求生产级性能优化
4. 不重写前端页面结构

---

## 四、约束与风险

1. 必须优先遵守冻结接口文档、调度模块输入输出约定和系统验收规范
2. 批量模拟结果结构不能再增删字段
3. 当前存在历史快照文档，需要避免把旧口径继续带回实现
4. `NO_SHOW` 与故障重排若处理不当，最容易重新破坏接口一致性
