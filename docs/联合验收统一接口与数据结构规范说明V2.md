# 联合验收统一接口与数据结构规范说明

> 文档用途：用于多个小组在联合验收前统一客户端、服务端、测试用例、详单账单、状态语义和验收指标  
> 适用场景：3 个及以上小组共同参加联合验收，随机抽取小组充当服务器端，其余小组充当客户端电动车
> 版本：V2.0
> 更新日期：2025-03-30 18:20

---

## 一、文档目标

联合验收并不要求各小组的内部实现完全一致，但必须保证：

- 外部接口一致
- 数据结构一致
- 状态语义一致
- 测试用例格式一致
- 输出结果格式一致

否则即使各组都“实现了自己的系统”，也无法完成跨组联调和统一验收。

因此，本规范的目标是明确：

1. 哪些功能设计必须统一
2. 哪些内部实现可以保留差异
3. 客户端与服务端如何通信
4. 单次模拟和批量模拟分别如何设计
5. 详单、账单和评测指标如何统一

---

## 二、联合验收中的统一原则

### 2.1 必须统一的内容

以下内容必须统一，否则联合验收无法正常开展：

1. 核心业务流程
2. 客户端与服务端的接口定义
3. 请求和响应字段名称
4. 主要状态和异常状态的含义
5. 批量模拟测试用例的数据结构
6. 详单和账单的输出结构
7. 评测指标的定义方式

### 2.2 可以不统一的内容

以下内容可以由各组自行实现：

1. 调度算法内部实现
2. 前后端技术栈
3. 数据库表结构
4. 页面设计和交互风格
5. 日志实现方式

结论：

- 外部协议必须统一
- 内部实现允许不同

## 三、必须统一的功能设计

这一部分非常关键。因为有些小组可能尚未完全明确需求，但联合验收要求大家至少在“对外可协作的系统能力”上达成一致。

### 3.1 必须统一的服务端功能

每个可充当服务器端的小组，至少必须具备以下功能：

1. 受理充电请求
2. 将用户纳入排队或调度流程
3. 返回当前状态与预计等待时长
4. 发起叫号提醒
5. 处理用户到场确认
6. 处理开始充电
7. 处理充电完成
8. 发起挪车提醒
9. 生成详单
10. 生成账单
11. 处理支付确认
12. 输出基础统计结果
13. 处理申报量与实际量不一致
14. 处理异常情况

异常情况至少包括：

1. 用户取消排队
2. 用户过号未到
3. 用户中断充电
4. 用户未及时挪车
5. 充电桩故障

说明：

- 如果某个小组没有实现这些核心功能，即使能本地演示，也无法在联合验收中扮演标准服务器端
- 因此，这些功能属于“联合验收最低公共能力”

### 3.2 必须统一的客户端功能

每个可充当客户端的小组，至少必须具备以下功能：

1. 提交单个充电请求
2. 查询当前请求状态与预计等待时长
3. 接收叫号提醒
4. 发送到场确认
5. 处理取消排队
6. 处理中断充电
7. 处理挪车确认
8. 接收详单和账单
9. 完成支付确认
10. 支持批量模拟上传

说明：

- 客户端不一定需要界面完全一致
- 但客户端必须能表达统一的事件和状态
- 否则不同组的客户端无法驱动别组的服务器端

### 3.3 为什么必须统一功能设计

如果不统一功能设计，会出现以下问题：

1. 有的服务端不支持“过号”
2. 有的客户端没有“确认到场”事件
3. 有的组只输出账单，不输出详单
4. 有的组不处理“未挪车”
5. 有的组不返回预计等待时长
6. 有的组不支持支付确认
7. 有的组不处理申报量与实际量差异
8. 有的组不支持批量模拟

这会导致：

- 测试用例无法复用
- 服务端与客户端语义不一致
- 联合验收无法公平比较
- 结果无法统一汇总

因此，联合验收下，“功能设计”至少要在核心验收能力上统一。

## 四、统一的业务流程

### 4.1 单次模拟标准流程

联合验收中，单次模拟建议统一为以下完整流程：

1. 用户提交请求
2. 用户进入排队
3. 服务端返回当前状态和预计等待时间
4. 服务端叫号提醒
5. 用户确认到场
6. 开始充电
7. 若实际充电量提前达到车辆上限，则提前结束充电
8. 充电结束
9. 服务端提醒挪车
10. 用户确认挪车
11. 服务端生成详单和账单
12. 用户完成支付确认

### 4.2 单次模拟中的异常分支

在标准流程之外，应统一支持以下分支：

1. 用户取消排队
2. 用户过号未到
3. 用户中断充电
4. 用户未及时挪车
5. 充电桩故障
6. 用户申报量与实际充电量不一致

说明：

- 这些分支不要求每次测试都触发
- 但接口和状态定义必须预留

### 4.3 必须显式纳入公共功能的能力

以下能力虽然容易被理解成“实现细节”，但在联合验收中必须显式视为公共功能：

1. 查看预计等待时长  
   说明：
   客户端不能只看到 `WAITING` 等状态，还必须能获取服务端返回的预计等待时长。

2. 临近确认机制  
   说明：
   在本规范中，临近确认机制等价于“叫号提醒 + 用户确认到场”。

3. 申报量与实际量差异处理  
   说明：
   `request_energy` 用于预测，`actual_energy` 用于结算。该能力必须统一，否则详单和账单无法解释。

4. 通知提醒  
   说明：
   至少包括叫号提醒和挪车提醒；如果界面实现不同，也必须保留同样的业务语义。

5. 支付确认  
   说明：
   即使采用模拟支付，也必须在流程和账单结果中体现支付状态。

6. 基础统计输出  
   说明：
   批量模拟结果中必须返回平均等待时间、平均完成时间、充电桩利用率等基础统计指标。

## 五、通信方式规范

### 5.1 推荐通信方案

建议统一采用：

- 同一局域网通信
- HTTP/HTTPS 接口
- JSON 数据格式

说明：

- 不建议每组自定义 socket 协议
- HTTP + JSON 最容易在 mac 和 Windows 之间联调
- 客户端可以是网页、桌面程序或命令行程序，但对外都应使用统一接口

### 5.2 服务端访问方式

服务端应支持局域网访问，例如：

`http://<server_ip>:<port>`

服务端不得只监听 `127.0.0.1`，应监听可被局域网访问的地址。

### 5.3 跨设备通信的统一实现要求

为保证 mac 和 Windows 设备能够互相访问，建议各组统一满足以下要求：

1. 三台电脑必须接入同一局域网  
   可以是同一个校园 Wi-Fi、同一个路由器，或同一部手机开的热点。

2. 服务端程序必须监听局域网地址  
   不要只监听 `127.0.0.1` 或 `localhost`，应监听 `0.0.0.0` 或指定本机局域网 IP。

3. 客户端必须通过局域网 IP 访问服务端  
   例如：
   `http://192.168.1.23:8080`

4. 必须预留一个健康检查接口  
   例如：
   `GET /health`

5. 所有组统一约定端口号或至少约定端口范围  
   例如统一为 `8080`、`9000` 等，避免联调时混乱。

### 5.4 mac 作为服务器端的配置说明

当 mac 充当服务器端时，建议按以下步骤检查：

#### 第一步：确认已连接同一局域网

- 打开“系统设置”
- 进入“网络”
- 确认当前 Wi-Fi 或 Ethernet 已连接

#### 第二步：查看本机局域网 IP

根据 Apple 的网络设置说明，可在：

- `系统设置 -> 网络 -> 当前网络 -> 详情 -> TCP/IP`

查看当前 IPv4 地址。这个地址就是其他客户端访问服务端时应使用的地址。

#### 第三步：确认防火墙未阻止服务端程序

根据 Apple 的防火墙设置说明：

- 打开 `系统设置 -> 网络 -> 防火墙`
- 如果已开启防火墙，进入“选项”
- 将你们的服务端程序加入“允许传入连接”的列表
- 不要开启“阻止所有传入连接”

#### 第四步：确认本地网络权限

根据 Apple 的本地网络权限说明：

- 打开 `系统设置 -> 隐私与安全性 -> 本地网络`
- 确认浏览器、终端、IDE 或你们的服务端程序已被允许访问本地网络

#### 第五步：启动服务端

启动服务端时，应确保服务监听在：

- `0.0.0.0:<port>`

而不是：

- `127.0.0.1:<port>`

否则别的电脑无法访问。

### 5.5 mac 作为客户端的配置说明

当 mac 充当客户端时，建议按以下步骤检查：

1. 确认与服务器端处于同一局域网
2. 确认客户端程序具备本地网络权限
3. 在浏览器、终端或客户端程序中使用服务器局域网 IP 访问接口
4. 优先访问健康检查接口进行联通测试

例如：

`http://192.168.1.23:8080/health`

### 5.6 Windows 作为服务器端的配置说明

当 Windows 充当服务器端时，建议按以下步骤检查：

#### 第一步：确认网络为 Private

根据 Microsoft 的网络设置说明，Windows 的网络分为 Public 和 Private。

联合验收时，建议将当前局域网设置为：

- `Private`

因为在 Public 网络下，设备更倾向于隐藏自己，不利于其他设备访问服务端。

操作路径建议为：

- `设置 -> 网络和 Internet -> 当前网络 -> 网络配置文件类型 -> Private`

#### 第二步：查看本机局域网 IP

根据 Microsoft 的网络设置说明，可在：

- `设置 -> 网络和 Internet -> 当前网络 -> 属性`

查看本机 IPv4 地址。

#### 第三步：允许服务端程序通过防火墙

根据 Microsoft 的防火墙说明：

- 打开 `Windows 安全中心 -> 防火墙和网络保护`
- 选择“允许应用通过防火墙”
- 将你们的服务端程序加入允许列表
- 至少勾选 `Private` 网络

更推荐“允许应用”，不推荐直接长期开放任意端口。

#### 第四步：启动服务端

启动服务端时，应确保监听：

- `0.0.0.0:<port>`

而不是：

- `127.0.0.1:<port>`

### 5.7 Windows 作为客户端的配置说明

当 Windows 充当客户端时，建议按以下步骤检查：

1. 确认当前网络与服务器处于同一局域网
2. 确认客户端应用未被防火墙拦截
3. 使用服务器的局域网 IP 访问服务端
4. 先访问健康检查接口，再访问正式业务接口

### 5.8 联调前统一排查步骤

建议三组在正式联合验收前，统一进行以下排查：

1. 服务器端打印本机 IP 和监听端口
2. 客户端使用浏览器访问 `http://<server_ip>:<port>/health`
3. 若访问失败，依次检查：
   - 是否处于同一局域网
   - 服务端是否监听 `0.0.0.0`
   - 防火墙是否允许程序
   - mac 是否授予本地网络权限
   - Windows 是否设置为 Private 网络
4. 健康检查通过后，再测试业务接口

## 六、统一的核心数据结构

## 6.1 统一字段命名原则

建议全部采用英文小写加下划线命名，避免不同组之间命名风格不一致。

例如：

- `request_id`
- `charge_mode`
- `request_energy`
- `estimated_wait_seconds`

### 6.2 基础枚举定义

#### 6.2.1 充电模式

```json
"FAST"
"SLOW"
```

中文说明：

- `FAST`：快充
- `SLOW`：慢充

#### 6.2.2 请求状态

```json
"PENDING"
"WAITING"
"CALLED"
"CONFIRMED"
"CHARGING"
"COMPLETED"
"COMPLETED_EARLY"
"CANCELLED"
"NO_SHOW"
"INTERRUPTED"
"FAULT_REQUEUE"
```

中文说明：

- `PENDING`：请求已提交，尚未进入正式预测
- `WAITING`：正在等待调度
- `CALLED`：已被叫号，等待用户确认到场
- `CONFIRMED`：用户已确认到场
- `CHARGING`：正在充电
- `COMPLETED`：正常充电完成
- `COMPLETED_EARLY`：提前充满并结束
- `CANCELLED`：已取消排队
- `NO_SHOW`：过号未到
- `INTERRUPTED`：中断充电
- `FAULT_REQUEUE`：因故障重新入队

#### 6.2.3 充电桩状态

```json
"IDLE"
"RESERVED"
"CHARGING"
"WAITING_TO_LEAVE"
"FAULT"
```

中文说明：

- `IDLE`：空闲
- `RESERVED`：已预留，等待用户接入
- `CHARGING`：正在充电
- `WAITING_TO_LEAVE`：充电结束，等待挪车
- `FAULT`：故障，不可用

#### 6.2.4 支付状态

```json
"UNPAID"
"PAID"
"FAILED"
```

中文说明：

- `UNPAID`：未支付
- `PAID`：已支付
- `FAILED`：支付失败

## 七、单次模拟接口规范

### 7.1 提交充电请求

#### 请求

`POST /api/request/create`

```json
{
  "request_time": "2026-03-30T10:00:00",
  "charge_mode": "FAST",
  "request_energy": 20.0
}
```

字段说明：

| 字段 | 中文解释 |
|------|----------|
| `request_time` | 发起请求时间 |
| `charge_mode` | 充电模式，快充或慢充 |
| `request_energy` | 申请充电量 |

#### 响应

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "request_id": "REQ001",
    "status": "WAITING",
    "estimated_wait_seconds": 600,
    "estimated_start_time": "2026-03-30T10:10:00",
    "estimated_finish_time": "2026-03-30T11:10:00"
  }
}
```

字段说明：

| 字段 | 中文解释 |
|------|----------|
| `code` | 业务状态码，`0` 表示成功 |
| `message` | 返回信息 |
| `data.request_id` | 请求编号 |
| `data.status` | 当前请求状态 |
| `data.estimated_wait_seconds` | 预计等待秒数 |
| `data.estimated_start_time` | 预计开始充电时间 |
| `data.estimated_finish_time` | 预计完成时间 |

### 7.2 查询请求状态

#### 请求

`GET /api/request/status/{request_id}`

#### 响应

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "request_id": "REQ001",
    "status": "CALLED",
    "station_id": "FAST_01",
    "estimated_wait_seconds": 0,
    "last_called_time": "2026-03-30T10:08:00"
  }
}
```

字段说明：

| 字段 | 中文解释 |
|------|----------|
| `data.request_id` | 请求编号 |
| `data.status` | 当前状态 |
| `data.station_id` | 当前分配或预分配的充电桩编号 |
| `data.estimated_wait_seconds` | 当前剩余预计等待秒数 |
| `data.last_called_time` | 最近一次叫号时间 |

### 7.3 用户确认到场

#### 请求

`POST /api/request/confirm_arrival`

```json
{
  "request_id": "REQ001",
  "confirm_time": "2026-03-30T10:09:00"
}
```

字段说明：

| 字段 | 中文解释 |
|------|----------|
| `request_id` | 请求编号 |
| `confirm_time` | 用户确认到场时间 |

#### 响应

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "request_id": "REQ001",
    "status": "CONFIRMED",
    "station_id": "FAST_01"
  }
}
```

### 7.4 取消排队

#### 请求

`POST /api/request/cancel_queue`

```json
{
  "request_id": "REQ001",
  "cancel_time": "2026-03-30T10:05:00"
}
```

字段说明：

| 字段 | 中文解释 |
|------|----------|
| `request_id` | 请求编号 |
| `cancel_time` | 取消排队时间 |

### 7.5 中断充电

#### 请求

`POST /api/request/interrupt_charge`

```json
{
  "request_id": "REQ001",
  "interrupt_time": "2026-03-30T10:40:00"
}
```

字段说明：

| 字段 | 中文解释 |
|------|----------|
| `request_id` | 请求编号 |
| `interrupt_time` | 中断充电时间 |

### 7.6 确认挪车

#### 请求

`POST /api/request/confirm_leave`

```json
{
  "request_id": "REQ001",
  "leave_time": "2026-03-30T11:15:00"
}
```

字段说明：

| 字段 | 中文解释 |
|------|----------|
| `request_id` | 请求编号 |
| `leave_time` | 最终挪车确认时间 |

### 7.7 获取详单与账单

#### 请求

`GET /api/request/result/{request_id}`

#### 响应

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "request_id": "REQ001",
    "detail": {},
    "bill": {}
  }
}
```

### 7.8 支付确认

#### 请求

`POST /api/request/pay`

```json
{
  "request_id": "REQ001",
  "pay_time": "2026-03-30T11:20:00",
  "pay_amount": 27.75
}
```

字段说明：

| 字段 | 中文解释 |
|------|----------|
| `request_id` | 请求编号 |
| `pay_time` | 支付确认时间 |
| `pay_amount` | 本次支付金额 |

## 八、单次模拟统一详单结构

详单必须统一，因为这是联合验收最终比对的重要依据。

```json
{
  "request_id": "REQ001",
  "charge_mode": "FAST",
  "request_energy": 20.0,
  "actual_energy": 18.5,
  "request_time": "2026-03-30T10:00:00",
  "queue_enter_time": "2026-03-30T10:00:00",
  "called_time": "2026-03-30T10:08:00",
  "arrival_confirm_time": "2026-03-30T10:09:00",
```

- 联合验收不仅看“正常完成”

### 8.2 关于申报量与实际量差异的统一要求

- `request_energy`

- 若 `actual_energy < request_energy`，可表示提前充满、故障中断或用户主动中断后的实际结果
- 各组不能只保留其中一个字段，否则无法统一解释调度预测与最终结算的关系

## 十、批量模拟设计规范

### 10.1 为什么必须统一批量模拟

联合验收中，批量模拟是最容易出现分歧的部分。如果每组：

- 自己定义一套测试用例结构
- 自己理解一套事件触发方式
- 自己决定输出哪些字段

那么：

- 无法进行统一自动测试
- 无法比较不同服务器的结果
- 无法做稳定性和并发验证

因此，批量模拟必须统一。

### 10.2 推荐批量模拟方式

建议批量模拟采用：

**客户端一次性提交标准测试用例 JSON，服务端按时间顺序回放并执行调度，然后统一返回每个用户的详单和账单。**

这样做的好处是：

- 跨平台最稳定
- 不要求客户端实时持续联网发事件
- 批量结果可重复
- 更容易比较不同服务器的调度效果

### 10.3 批量测试输入结构

#### 请求

`POST /api/batch/simulate`

```json
{
  "test_case_id": "BATCH_001",
  "users": [
    {
      "user_id": "U001",
      "request_time": "2026-03-30T10:00:00",

      "request_energy": 20.0,
```

- `summary` 属于公共能力中的“基础统计输出”
- 各组至少必须返回平均等待时间、平均完成时间、充电桩利用率这三个指标

字段说明：

### 10.6 批量模拟中的并发处理建议

联合验收中批量模拟可能涉及大量用户请求，因此服务端至少应保证：

1. 同一时刻事件处理顺序可重复
2. 同一个充电桩不会被重复分配
3. 队列更新和桩状态更新具有一致性

推荐做法：

- 服务端按 `request_time` 对事件排序回放
- 若同一时刻多个事件冲突，则按统一优先级处理

建议统一事件处理优先级：

1. 故障事件
2. 取消事件
3. 过号判定
4. 到场确认
5. 开始充电
6. 结束充电
7. 挪车确认
8. 新请求进入

说明：

- 不要求每组必须用相同的线程模型
- 但必须保证同样的测试用例输入能得到语义一致的输出

## 十一、调度算法评测指标统一建议

### 11.1 为什么必须统一评测指标

如果各组：

- 有的看平均等待时间
- 有的看平均完成时间
- 有的只看利用率

那么调度效果无法比较。

因此，建议统一指标体系。

### 11.2 推荐指标

参考操作系统进程调度中的评测标准，建议采用：

#### 主指标

- 平均完成时间  
  `avg_finish_time = 平均(等待时间 + 实际充电时间)`

#### 辅指标

- 平均等待时间
- 最大等待时间
- 充电桩利用率
- 吞吐量

### 11.3 指标定义

#### 平均等待时间

`avg_wait_time = 所有用户等待时间总和 / 用户总数`

#### 平均完成时间

`avg_finish_time = 所有用户完成时间总和 / 用户总数`

其中：

`完成时间 = 排队等待时间 + 实际充电时间`

#### 充电桩利用率

`station_utilization = 总充电桩忙碌时长 / 总可用时长`

#### 吞吐量

`throughput = 单位时间内完成充电的用户数`

### 11.4 建议结论

联合验收中建议：

- 以**平均完成时间**作为主评价指标
- 以**平均等待时间**和**充电桩利用率**作为辅助指标

原因：

- 平均完成时间能同时反映排队和充电两个阶段
- 平均等待时间能体现题目中“排队时间最短”的要求
- 利用率能体现系统整体资源利用水平

## 十二、统一错误响应格式

所有接口建议统一返回：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

其中：

- `code = 0` 表示成功
- 非 `0` 表示失败

建议统一错误码：

| code | 含义 |
|------|------|
| 0 | 成功 |
| 1001 | 请求参数错误 |
| 1002 | 请求不存在 |
| 1003 | 当前状态不允许该操作 |
| 1004 | 过号，无法确认到场 |
| 1005 | 已取消，无法继续操作 |
| 1006 | 充电桩故障 |
| 1007 | 支付失败 |
| 1099 | 服务器内部错误 |

## 十四、结论

联合验收下，三个小组的内部实现不必完全一致，但以下内容必须高度统一：

- 核心功能设计
- 接口流程
- 数据结构
- 状态语义
- 批量测试格式
- 详单账单格式
- 评测指标

可以把这份文档理解为“联合验收的最小公共协议”。  
只要三组都遵守这份协议，就算各自内部调度算法、架构设计不同，也能够顺利完成联合验收。
