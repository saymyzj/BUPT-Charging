# 系统验收接口与测试输入规范说明

> 说明：内容已根据老师最终要求改为“系统验收”口径  
> 适用场景：本组系统在最终验收时接受老师给出的单机测试、批量测试和展示验证  
> 版本：V3.0  
> 更新日期：2026-04-01

---

## 一、文档目标

本规范用于统一以下内容：

1. 单次请求接口的字段与状态语义
2. 批量测试输入结构
3. 自然语言验收条件到结构化输入的映射规则
4. 详单、账单和评测指标的输出口径
5. 队列长度歧义下的兼容处理方式

本规范不要求内部实现完全一致，但要求：

- 外部字段命名稳定
- 状态语义稳定
- 同一测试输入能够得到语义一致的输出
- 统计指标的定义方式稳定

## 三、系统验收的核心业务流程

### 3.1 单次流程

系统验收中的标准流程如下：

1. 用户提交请求
2. 系统返回当前状态和预计等待时间
3. 用户进入等待池或桩级队列对应的内部状态
4. 服务端发起叫号提醒
5. 用户确认到场
6. 开始充电
7. 若提前充满，则提前结束
8. 充电结束
9. 服务端提醒挪车
10. 用户确认挪车
11. 系统生成详单和账单
12. 用户完成支付确认

### 3.2 异常分支

最终验收中应支持以下异常分支：

1. 用户取消排队
2. 用户过号未到
3. 用户中断充电
4. 用户未及时挪车
5. 充电桩故障
6. 用户申报量与实际量不一致

## 四、验收场景参数

### 4.1 必备场景参数

最终验收中，以下参数应视为场景输入，而不是固定常量：

| 字段 | 含义 |
|------|------|
| `fast_station_count` | 快充桩数量 |
| `slow_station_count` | 慢充桩数量 |
| `waiting_area_capacity` | 等待区车位总容量，由快充池和慢充池共享 |
| `station_queue_mode` | 队列输入解释方式 |

补充说明：

- `waiting_area_capacity` 是系统全局等候区容量
- 快充等待池和慢充等待池共同占用该容量，而不是分别拥有独立容量

### 4.2 队列输入的两种兼容方式

老师确认“充电桩排队队列长度”可能有两种含义，系统必须兼容：

#### 方式 A：统一容量模式

若题目只给出一个“充电桩排队队列长度”，则解释为：

- 每个充电桩的统一排队容量上限
- 对应字段：
  - `station_queue_mode = "UNIFORM_CAPACITY"`
  - `station_queue_capacity = <整数>`

#### 方式 B：桩级快照模式

若题目直接给出“某桩正在充电、有几人排队、某桩空闲”等描述，则解释为：

- 每个充电桩的初始状态快照
- 对应字段：
  - `station_queue_mode = "STATION_SNAPSHOT"`
  - `station_snapshots = [...]`

每个 `station_snapshot` 建议包含：

| 字段 | 含义 |
|------|------|
| `station_code` | 桩编号 |
| `station_type` | `FAST` / `SLOW` |
| `status` | `IDLE` / `CHARGING` / `WAITING_TO_LEAVE` / `FAULT` |
| `current_user_remaining_seconds` | 当前充电用户剩余时长，可空 |
| `queue_length` | 当前桩后已排队人数 |

### 4.3 内部兼容原则

无论采用哪种输入方式，系统都应先完成场景适配：

1. 初始化桩数量和桩类型
2. 初始化等待区容量
3. 将桩级快照或统一容量参数转换为内部状态
4. 再交给统一调度器执行

## 五、批量测试推荐输入结构

### 5.1 请求

`POST /api/batch/simulate`

```json
{
  "test_case_id": "BATCH_001",
  "scenario": {
    "fast_station_count": 2,
    "slow_station_count": 3,
    "waiting_area_capacity": 6,
    "station_queue_mode": "UNIFORM_CAPACITY",
    "station_queue_capacity": 3
  },
  "users": [
    {
      "user_id": "U001",
      "request_time": "2026-03-30T10:00:00",
      "charge_mode": "FAST",
      "request_energy": 20.0,
      "cancel_queue": false,
      "cancel_time": null,
      "confirm_arrival_delay_seconds": 60,
      "interrupt_charge": false,
      "interrupt_time": null,
      "leave_delay_seconds": 120
    }
  ]
}
```

### 5.2 桩级快照模式示例

```json
{
  "test_case_id": "BATCH_002",
  "scenario": {
    "fast_station_count": 2,
    "slow_station_count": 1,
    "waiting_area_capacity": 8,
    "station_queue_mode": "STATION_SNAPSHOT",
    "station_snapshots": [
      {
        "station_code": "FAST_01",
        "station_type": "FAST",
        "status": "CHARGING",
        "current_user_remaining_seconds": 900,
        "queue_length": 3
      },
      {
        "station_code": "FAST_02",
        "station_type": "FAST",
        "status": "IDLE",
        "current_user_remaining_seconds": null,
        "queue_length": 0
      }
    ]
  },
  "users": []
}
```

### 5.3 字段说明

| 字段 | 中文解释 |
|------|----------|
| `test_case_id` | 批量测试用例编号 |
| `scenario` | 本次场景参数 |
| `users` | 本次批量测试的用户集合 |
| `request_time` | 用户提交请求时间 |
| `charge_mode` | 充电模式 |
| `request_energy` | 申请充电量 |
| `cancel_queue` | 是否取消排队 |
| `cancel_time` | 取消时间 |
| `confirm_arrival_delay_seconds` | 收到叫号后延迟多少秒确认到场 |
| `interrupt_charge` | 是否中断充电 |
| `interrupt_time` | 中断时间 |
| `leave_delay_seconds` | 充电完成后延迟多少秒挪车 |

### 5.4 自然语言到结构化输入的映射建议

若老师先给出自然语言测试条件，可按以下顺序映射：

1. 识别快充桩数量和慢充桩数量
2. 识别等待区车位容量
3. 判断“队列长度”属于统一容量还是桩级快照
4. 生成 `scenario`
5. 再把车辆请求逐条写入 `users`

## 六、批量测试输出结构

### 6.1 推荐响应

```json
{
  "test_case_id": "BATCH_001",
  "summary": {
    "total_users": 10,
    "completed_users": 9,
    "avg_wait_seconds": 320,
    "avg_finish_seconds": 1850,
    "total_finish_seconds": 16650,
    "station_utilization": 0.81
  },
  "results": [
    {
      "user_id": "U001",
      "detail": {},
      "bill": {}
    }
  ]
}
```

### 6.2 `summary` 字段说明

| 字段 | 含义 |
|------|------|
| `summary.total_users` | 用户总数 |
| `summary.completed_users` | 完成流程用户数 |
| `summary.avg_wait_seconds` | 平均等待秒数 |
| `summary.avg_finish_seconds` | 平均完成秒数 |
| `summary.total_finish_seconds` | 所有用户完成时间总和 |
| `summary.station_utilization` | 充电桩利用率 |

## 七、评测指标统一建议

### 7.1 主指标

最终验收中建议以：

- `avg_finish_seconds`

作为主指标，因为它直接对应“单用户本次总时长 = 等待 + 充电”的要求。

### 7.2 辅指标

建议同时输出：

- `avg_wait_seconds`
- `station_utilization`
- `total_finish_seconds`
- `max_wait_seconds`（可选）

其中：

- `avg_wait_seconds` 体现等待时间优化
- `station_utilization` 体现资源利用率
- `total_finish_seconds` 对应扩展要求中的系统整体总时长

---

## 八、结论

当前版本的规范重点不再是多组联合验收，而是：

1. 支持最终验收的灵活场景参数
2. 兼容两种“队列长度”解释
3. 保持接口、状态和输出口径稳定
4. 用统一指标解释调度效果

可以把这份文档理解为“系统验收下的最小公共协议”。
