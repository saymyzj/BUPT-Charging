# 智能充电桩调度计费系统：系统验收接口与测试输入规范说明

> 版本：V3.0
> 状态：最终冻结
> 冻结日期：2026-04-19
> 维护人：组长A
> 目标：统一系统验收、批量测试、部署验收和场景回放输入结构

## 一、验收总原则

系统验收以最终要求为唯一标准，重点验证：

1. 严格的等候区 + 每桩固定队列模型
2. `F/T` 排队号与顺序语义
3. 普通调度、故障调度、扩展调度的模式边界
4. 峰平谷计费与服务费
5. 管理端控制能力
6. 正式部署上线后的访问能力

## 二、验收场景参数

以下参数均视为正式场景输入，而不是代码常量：

| 字段 | 含义 |
|------|------|
| `fast_station_count` | 快充桩数量 |
| `slow_station_count` | 慢充桩数量 |
| `waiting_area_capacity` | 等候区容量 |
| `charging_queue_len` | 每桩固定队列长度，包含充电位 |
| `dispatch_mode` | `NORMAL` / `EXT_SINGLE_BATCH` / `EXT_FULL_BATCH` |
| `fault_dispatch_mode` | `PRIORITY` / `TIME_ORDER` |

## 三、单次业务主线流程

验收时必须可观察到以下完整主线（任何一步缺失或错序视为不通过）：

```
用户提交请求 → 分配 F/T 排队号 → 进入等候区 → 等候区叫号（取头车） →
入桩队列（按普通调度规则选桩） → 队首开始充电 → 充电完成 → 生成详单
```

对应状态转移：`WAITING_AREA → QUEUED → CHARGING → COMPLETED`。

## 四、验收异常分支清单

逐条覆盖最终要求 6/7 的异常动作，验收必须对每条分支至少走一遍：

| 编号 | 分支 | 预期终态 | 预期详单 |
|------|------|----------|----------|
| B1 | 等候区修改充电模式 | 重新生成 `F/T` 排队号，回到等候区尾部 | 不生成 |
| B2 | 等候区修改请求电量 | 保留排队号，更新请求电量 | 不生成 |
| B3 | 等候区取消请求 | `CANCELLED` | 不生成 |
| B4 | 充电区取消请求（= 提前结束并结算） | `COMPLETED_EARLY` | 生成，`actual_energy` 截至停止时刻 |
| B5 | 充电桩故障 + `PRIORITY` | 故障队列优先补位 | 被中断车生成 `FAULT_INTERRUPTED` 详单 |
| B6 | 充电桩故障 + `TIME_ORDER` | 同类型未开始充电车辆按排队号重排 | 被中断车生成 `FAULT_INTERRUPTED` 详单 |
| B7 | 故障恢复重排 | 固定按时间顺序重排 | 不额外生成 |

具体场景剧本见 [故障与扩展场景指南.md](故障与扩展场景指南.md)。

## 六、扩展模式测试要求

### 6.1 `EXT_SINGLE_BATCH`

测试输入必须明确：

- 当前调度模式是 `EXT_SINGLE_BATCH`
- 同时存在多个空位
- 本次联合调度只在模式内优化

### 6.2 `EXT_FULL_BATCH`

测试输入必须明确：

- 当前调度模式是 `EXT_FULL_BATCH`
- 到站车辆数等于总车位数
- 当前批次不区分快慢充

## 七、故障测试要求

### 7.1 故障发生

验收需显式说明：

- 故障桩编号
- 故障发生时间
- 当前故障策略
- 故障前该桩是否有排队车辆

### 7.2 故障恢复

验收需显式说明：

- 恢复时间
- 恢复时其它同类型桩中是否仍有未开始充电车辆

## 八、验收输出重点

### 8.1 用户端输出

必须能看到：

- `queue_number`
- `front_waiting_count`
- `request_status`
- `station_code`
- `estimated_wait_seconds / estimated_start_time / estimated_finish_time`
- 详单核心字段

### 8.2 管理端输出

必须能看到：

- 桩状态
- 桩队列内车辆信息
- 调度模式
- 故障策略
- 报表字段
- 用户/车辆信息维护（F-A07）列表与电池容量编辑

### 8.3 计费输出

账单至少应可拆分出：

- `charge_fee`
- `service_fee`
- `total_fee`

详单/账单/报表字段字典以 [冻结接口文档.md](冻结接口文档.md) §9/§10/§11 为准。

## 九、部署上线验收

系统上线验收除业务功能外，还应验证：

1. 公网 HTTPS 域名可访问
2. 前端页面可正常加载
3. `/api` 请求通过同域反代成功访问后端
4. SQLite 数据持久化正常
5. 发布前备份和回滚步骤存在并可执行
6. 当前部署方案不依赖 OSS
7. clouDNS 域名解析生效
8. 后端服务可通过 `systemd` 正常管理

完整 checklist 见 [部署上线手册.md](部署上线手册.md) §十。

## 十、最终判定口径

以下情况视为不通过：

- 系统仍以中央等待池作为正式运行模型
- 慢充默认功率仍写为 `7`
- 账单口径仍以旧 `energy_fee / time_fee / occupancy_fee` 为主
- 故障恢复未按时间顺序重排
- 扩展模式与普通模式边界不清
- 文档未覆盖正式部署上线口径
- 充电区取消最终态未落到 `COMPLETED_EARLY`（任一处仍写为 `CANCELLED` 视为不通过）
- 详单 / 账单 / 报表缺失 [冻结接口文档.md](冻结接口文档.md) §9/§10/§11 字段字典中任一条目
- `F-U / F-A / F-J` 编号与 [功能设计.md](功能设计.md) §一 编号表不一致
