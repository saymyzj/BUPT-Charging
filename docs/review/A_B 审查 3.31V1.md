## 2.1 `create_request` 与 `get_status` 的状态和预计时间口径不一致

### 问题描述

当前在 `backend/app/routes/request.py` 中：

1. `create_request` 插入数据库时把请求状态写成了 `PENDING`
2. 但接口响应里直接返回了 `WAITING`
3. `estimated_wait_seconds`、`estimated_start_time`、`estimated_finish_time` 目前是写死的模拟值
4. `get_status` 又读取数据库真实状态，因此可能出现：
   - 创建接口返回 `WAITING`
   - 状态查询接口却查到 `PENDING`

这会导致：

- 前后端状态不一致
- 联合验收客户端看到的状态可能不一致
- 和冻结接口文档的语义不一致

### 你需要怎么改

你有两种可选方案，但建议采用方案A。

#### 方案A：严格对齐冻结接口文档

1. 请求创建后，应真正进入可展示的等待状态
2. 数据库中的状态也应更新到 `WAITING`
3. 预测时间字段至少应有统一来源，不能长期写死

即使调度模块还没接进来，也可以先：

- 在创建请求后写入一组“临时预测值”
- 同时把状态写为 `WAITING`

这样至少保证：

- 数据库状态
- `create` 接口响应
- `status` 接口响应

三者一致。

#### 方案B：保留 `PENDING`

如果你坚持创建后先保留 `PENDING`，那就必须同步修改：

1. `create_request` 响应
2. `get_status` 逻辑
3. 相关实现文档
4. 冻结接口文档

但这会牵涉更多对外口径修改，不推荐。

### 建议结论

请采用方案A。
