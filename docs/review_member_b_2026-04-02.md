## 主要问题

### 1. 进度文档和实现版冻结文档互相打架

问题：

- `backend/README_服务端实现进度.md` 里把 `result/pay` 写成已完成，把 `batch` 写成“框架完成”
- `backend/冻结接口文档_服务端实现版.md` 前面又写 `result/pay/batch` 待实现，后面接口清单又把 `result/pay` 标成已实现

结论：

- 目前这两份文档不能同时作为可信依据
- 必须统一为“已实现 / 部分实现 / 未实现”三种状态

### 2. 实现版冻结文档仍然沿用旧口径

问题：

- 仍然写“中期验收前不再修改”
- 仍然把当前阶段表述成旧的冻结模式

结论：

- 这和当前正式文档的“最终验收 / 系统验收”口径不一致
- 文档必须改成当前项目口径

### 3. `/health` 返回格式和正式冻结文档不一致

问题：

- 正式冻结文档中，`GET /health` 响应是裸结构 `{status, timestamp}`
- 当前代码返回统一包装 `{code, message, data}`

结论：

- 这里必须二选一：
  1. 改代码
  2. 或经组长确认后改正式冻结文档

当前不能两边都保留

### 4. `request_time` 处理和正式规范不一致

问题：

- 当前 `create_request` 使用服务器当前时间覆盖传入的 `request_time`

影响：

- 会影响批量模拟的时间排序
- 会影响详单时间线
- 会影响后续真实回放

结论：

- 这项必须纳入本轮修正清单

### 5. 批量模拟目前只是框架，不是完整实现

问题：

- `batch_simulate.py` 里仍然是基础顺序模拟
- 单用户模拟还是占位返回
- `summary` 指标全是 0

结论：

- 如果在进度文档里写成“框架完成”，这是合理的
- 但不能在实现版冻结文档里暗示已经达到正式规范要求

### 6. 测试文件目前更像草稿，不足以证明功能可用

问题：

- `backend/tests/test_scenario_adapter.py` 和当前 `db.py` 的初始化方式不匹配
- 依赖 Flask app context 的代码，测试里没有按同样方式搭环境

结论：

- 当前可以说“已补测试文件”
- 但不能说“测试已跑通”
