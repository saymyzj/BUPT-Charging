# 调度模块输入输出约定

> 文档用途：冻结组内“后端服务 <-> 调度模块”之间的最小输入输出与调用时机约定  
> 适用范围：成员A、成员B在实现预测排程、事件重调度、批量模拟回放时必须共同遵守  
> 说明：本文件冻结的是模块边界和核心语义，不冻结具体算法公式实现细节

---

## 一、冻结范围

本文冻结以下内容：

1. 调度模块的调用时机
2. 调度模块的最小输入
3. 调度模块的最小输出
4. `request_time` 的最终语义
5. 两种队列输入解释到内部状态的转换规则
6. 共享等待区容量的统计规则
7. 批量模拟在当前阶段的最小完成标准

本文不冻结以下内容：

1. 具体优先级公式的参数调优
2. 调度模块内部的数据结构实现
3. 增量重算还是全量重算的优化策略

## 二、调用时机约定

### 2.1 创建请求时立即预测

用户调用 `POST /api/request/create` 后，服务端必须立即调用一次调度模块预测接口。

该预测至少返回：

- `estimated_wait_seconds`
- `estimated_start_time`
- `estimated_finish_time`

### 2.2 关键事件发生后统一触发重调度

以下事件均视为关键事件，服务端处理完成后必须触发一次重调度：

1. 新请求进入
2. 用户取消排队
3. 用户确认叫号
4. 用户过号
5. 用户开始充电
6. 用户中断充电
7. 充电完成
8. 用户提前充满
9. 用户超时未挪车
10. 充电桩故障
11. 充电桩故障恢复
12. 用户完成挪车

### 2.3 批量模拟采用逐事件回放

批量模拟中：

1. 服务端先根据 `scenario` 初始化内部状态
2. 所有事件按 `request_time` 排序回放
3. 同一时刻按固定优先级处理
4. 每处理完一个事件，立即触发一次重调度 / 重预测

当前阶段允许：

- 先采用全量重算
- 不强制要求增量重算优化

## 三、最小输入约定

### 3.1 预测接口最小输入

服务端调用“单请求预测”时，至少提供以下信息：

| 字段 | 含义 |
|------|------|
| `request_id` | 请求编号 |
| `request_time` | 用户提交请求时间 |
| `charge_mode` | `FAST` / `SLOW` |
| `request_energy` | 申请充电量 |
| `scenario` | 当前场景参数 |
| `waiting_pools` | 当前等待池状态 |
| `stations` | 当前充电桩状态 |
| `now` | 本次预测基准时间 |

### 3.2 重调度接口最小输入

服务端调用“事件重调度”时，至少提供：

| 字段 | 含义 |
|------|------|
| `event_type` | 当前触发事件类型 |
| `event_time` | 当前事件时间 |
| `scenario` | 当前场景参数 |
| `waiting_pools` | 当前等待池状态 |
| `stations` | 当前充电桩状态 |
| `requests` | 当前所有未完成请求快照 |

### 3.3 批量模拟接口最小输入

服务端调用“批量模拟执行”时，至少提供：

| 字段 | 含义 |
|------|------|
| `test_case_id` | 测试编号 |
| `scenario` | 当前场景参数 |
| `users` | 全部用户行为配置 |
| `initial_state` | 场景适配后的内部初始状态 |

## 四、最小输出约定

### 4.1 单请求预测最小输出

调度模块对服务端至少返回：

| 字段 | 含义 |
|------|------|
| `request_id` | 请求编号 |
| `status` | 当前建议状态，通常为 `WAITING` |
| `waiting_pool_type` | `FAST_POOL` / `SLOW_POOL` |
| `queue_position` | 在对应等待池中的相对位置 |
| `estimated_wait_seconds` | 预计等待秒数 |
| `estimated_start_time` | 预计开始时间 |
| `estimated_finish_time` | 预计完成时间 |
| `recommended_station_id` | 推荐桩号，可为 `null` |

### 4.2 重调度最小输出

重调度后至少返回：

1. 所有受影响请求的最新预测结果
2. 是否需要重新叫号的标记
3. 若需要叫号，对应的目标请求和目标充电桩

### 4.3 批量模拟最小输出

批量模拟至少返回：

1. 每个用户的 `detail`
2. 每个用户的 `bill`
3. `summary.total_users`
4. `summary.completed_users`
5. `summary.avg_wait_seconds`
6. `summary.avg_finish_seconds`
7. `summary.total_finish_seconds`
8. `summary.station_utilization`

说明：

- 当前阶段可以暂不输出更复杂的中间事件流
- 但以上字段属于最小验收结果，不应缺失

---

## 五、`request_time` 语义约定

### 5.1 单次请求

在单次请求接口中：

- `request_time` 是业务提交时间
- 服务端必须校验其格式是否合法
- 服务端不得在正常路径中随意用服务器当前时间覆盖它

说明：

- 若后续确有需要记录服务器接收时间，应新增独立字段处理
- 不能用覆盖 `request_time` 的方式替代

### 5.2 批量模拟

在批量模拟中：

- `request_time` 是事件排序和回放的唯一基准时间
- 服务端必须严格按输入中的 `request_time` 排序

## 六、两种队列输入的内部转换规则

### 6.1 `UNIFORM_CAPACITY`

当 `scenario.station_queue_mode = "UNIFORM_CAPACITY"` 时：

1. `station_queue_capacity` 表示每个桩的统一排队容量上限
2. 所有桩初始状态默认为 `IDLE`
3. 所有桩初始 `available_time = now`
4. 初始等待池为空，由后续请求进入

### 6.2 `STATION_SNAPSHOT`

当 `scenario.station_queue_mode = "STATION_SNAPSHOT"` 时：

1. `station_snapshots` 表示每个桩的初始状态快照
2. 对于 `status = CHARGING` 且带有 `current_user_remaining_seconds` 的桩：
   - 初始 `available_time = now + current_user_remaining_seconds`
3. 对于快照中的 `queue_length`：
   - 视为该桩后已有待服务请求
   - 服务端需将其转换为内部初始等待请求集合
4. 转换完成后，统一进入 `FAST_POOL` / `SLOW_POOL` 参与后续预测和重调度

## 七、共享等待区容量统计规则

### 7.1 全局共享

`waiting_area_capacity` 是系统全局等候区总容量。

规则：

- 快充等待池和慢充等待池共享该容量
- 不是两个池分别独立拥有一个容量

### 7.2 计数对象

当前阶段统一规定：

以下请求计入等待区占用：

1. `WAITING` 状态的请求
2. 因快照转换而生成的初始待服务请求
3. `FAULT_REQUEUE` 后重新进入等待态的请求

以下状态不计入等待区占用：

1. `CONFIRMED`
2. `CHARGING`
3. `WAITING_TO_LEAVE`
4. `COMPLETED`
5. `COMPLETED_EARLY`
6. `CANCELLED`
7. `INTERRUPTED`
8. `NO_SHOW`

说明：

- `CALLED` 是否仍计入等待区占用，当前阶段统一按“不计入”处理
- 这样有利于成员B实现容量判断，也利于成员C展示状态变化

## 八、当前阶段完成标准

### 8.1 对成员B的最小要求

以下内容达到后，才可称为“本阶段后端主流程可审查”：

1. `/api/request/create` 能返回真实预测字段
2. `/api/request/status/{request_id}` 能反映当前状态和等待信息
3. 等待池容量判断符合共享等待区规则
4. 批量模拟接口能接收正式 `scenario + users` 结构
5. `summary` 字段齐全，即使算法仍可继续优化

### 8.2 对成员C的最小要求

以下内容达到后，才可称为“本阶段前端页面可审查”：

1. 登录、工作台、任务页、管理端总览可正常打开
2. 页面字段与冻结接口文档一致
3. 请求层对接当前冻结路径
4. 页面中涉及等待区容量、队列模式、summary 的展示口径与正式文档一致
