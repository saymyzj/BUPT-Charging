## 二、用户端功能

### 2.1 F-U01 注册与登录

- 注册必填字段：`username`、`password`、`battery_capacity`
- 用户名唯一；密码摘要保存，不保存明文
- 登录成功返回身份凭证，后续所有接口使用该凭证
- 角色枚举：`USER`、`ADMIN`
- 同一用户在同一时间最多只有一个未终态请求

### 2.2 F-U02 提交充电请求

用户在等候区提交：

- `charge_mode`（`FAST` / `SLOW`）
- `request_energy`（kWh，正数；接口硬性约束见 [冻结接口文档.md](冻结接口文档.md) §4 错误码 1008）

系统返回：

- `request_id`
- `queue_number`（`F1/F2/...` 或 `T1/T2/...`）
- `front_waiting_count`
- `request_status`

### 2.3 F-U03 查看本车排队信息

用户端必须展示：

- `queue_number`
- `charge_mode`
- `request_status`
- `front_waiting_count`（本模式下前车等待数量）
- `station_code`（若已进入某桩队列）
- `station_queue_position`（若已进入某桩队列）
- `estimated_wait_seconds`
- `estimated_start_time`
- `estimated_finish_time`

### 2.4 修改请求

#### 2.4.1 F-U04 修改充电模式

- 只允许在 `WAITING_AREA`
- 修改后重新生成排队号
- 新排队号插入新模式等候队尾
- 充电区不允许；如需改模式，只能先按 F-U07 结算退出，再以新请求重新提交

#### 2.4.2 F-U05 修改请求电量

- 只允许在 `WAITING_AREA`
- 排队号保持不变
- 充电区不允许；如需改电量，只能先按 F-U07 结算退出

### 2.5 取消或提前结束

#### 2.5.1 F-U06 等候区取消请求

- 允许状态：`WAITING_AREA`
- 成功后状态变为 `CANCELLED`
- 不生成详单
- 释放等候区名额

#### 2.5.2 F-U07 充电区取消 / 提前结束

最终要求 6.c"取消充电：等候区、充电区均允许"。本系统对充电区取消的落地语义定义为：

- 允许状态：`CHARGING` 或 `QUEUED`（已入桩队列但未开始充电）
- 本次充电过程按已充电量结算，生成独立详单
- 状态统一进入 `COMPLETED_EARLY`
- 充电区取消与充电区提前结束**语义等价**，共用接口 `POST /api/request/stop`
- 若需要继续充电，用户必须以新请求重新进入等候区

此设计与最终要求 7 中"故障中断产生独立详单"的精神一致——任意"充电过程的非正常提前结束"都对应一条独立详单。

### 2.6 F-U08 查看详单

详单字段清单与语义见 [冻结接口文档.md](冻结接口文档.md) §9 详单字段字典。至少包含：

- `detail_id`
- `detail_generated_at`
- `station_code`
- `actual_energy`
- `charge_duration_seconds`
- `start_time`
- `stop_time`
- `charge_fee`
- `service_fee`
- `total_fee`
- `request_status`

## 三、管理员端功能

### 3.1 F-A01 充电桩启动与关闭

- 管理员可启动处于 `SHUTDOWN` 的充电桩，状态转为 `RUNNING`
- 管理员可关闭空闲且队列为空的充电桩，状态转为 `SHUTDOWN`
- 正在服务或仍有排队车辆的桩不可直接关闭

> 此"空闲且队列为空"为本系统的设计决策，非最终要求显式规定。目的是避免关闭时遗留车辆导致的数据一致性问题。

### 3.2 F-A02 查看充电桩状态

管理员端必须展示：

- 当前是否正常工作（`station_status`）
- 系统启动后累计充电次数（`total_charge_count`）
- 充电总时长（`total_charge_seconds`）
- 充电总电量（`total_charge_energy`）
- 当前队列长度（`queue_length`）
- 当前服务车辆（`current_request_id`）

### 3.3 F-A03 查看单桩队列车辆信息

至少展示：

- 用户 ID（`user_id`）
- 车辆电池总容量（`battery_capacity`）
- 请求充电量（`request_energy`）
- 排队时长（`queue_wait_seconds`）
- 排队号码（`queue_number`）

### 3.4 F-A04 调度模式切换

管理员可在运行时切换：

- `NORMAL`
- `EXT_SINGLE_BATCH`
- `EXT_FULL_BATCH`

切换规则见 [调度算法.md](调度算法.md) §5。切换动作必须写入系统事件日志。

### 3.5 F-A05 故障策略切换

管理员可在运行时切换：

- `PRIORITY`
- `TIME_ORDER`

切换后，下一次故障发生时按新策略处理。

### 3.6 F-A06 故障标记与恢复

- 可手动标记单桩故障，状态转为 `FAULT`
- 若被标记的桩为空闲、且队列为空，则直接置 `FAULT`，不触发任何重调度
- 若被标记的桩有车在排队或正在充电，则按当前 `fault_dispatch_mode` 触发重调度
- 可手动恢复故障桩
- 恢复后若其它同类型桩仍有未开始充电车辆，必须按时间顺序重排（详见 [调度算法.md](调度算法.md) §3.4）

### 3.7 F-A07 用户与车辆信息维护

本模块对应最终要求 5.a"服务器端：用户（车辆）信息维护"。

展示与操作：

- 用户列表，每条展示：`user_id` / `username` / `battery_capacity` / `role` / `created_at`
- 查看单个用户的历史详单
- 修改车辆电池总容量：
  - 只允许在用户无活跃请求（即无 `WAITING_AREA / QUEUED / CHARGING` 状态请求）时执行
  - 修改记录进入系统事件日志

不包含：虚拟余额、扣费、注册审批、用户删除。这些能力不在最终要求范围内，不实现。

### 3.8 F-A08 报表展示

支持粒度：`day`、`week`、`month`（严格遵循最终要求 5.c，不支持其它粒度）。

展示字段：

- `time_key`（`YYYY-MM-DD` / `YYYY-Www` / `YYYY-MM`）
- `station_code`
- `total_charge_count`
- `total_charge_seconds`
- `total_charge_energy`
- `total_charge_fee`
- `total_service_fee`
- `total_fee`

字段语义与跨时段拆分规则见 [冻结接口文档.md](冻结接口文档.md) §11 报表字段字典。

## 四、计费功能

### 4.1 费用结构

- `total_fee = charge_fee + service_fee`
- `charge_fee = Σ(分时电价 × 对应时段内实际充电度数)`
- `service_fee = 0.8 × actual_energy`

### 4.2 电价时段

- 峰时：`10:00-15:00`、`18:00-21:00`，单价 `1.0`
- 平时：`07:00-10:00`、`15:00-18:00`、`21:00-23:00`，单价 `0.7`
- 谷时：`23:00-07:00`，单价 `0.4`

### 4.3 时长定义

- `charge_duration_hours = actual_energy / station_power_kw`
- 跨时段时按实际充电功率按秒线性切分到三个时段，分别计费后求和

## 五、故障处理功能（概要）

最终要求 7 的所有故障语义已在 [调度算法.md](调度算法.md) §3 给出完整算法，并在 [故障与扩展场景指南.md](故障与扩展场景指南.md) 给出验收场景剧本。本节仅保留一张接口级摘要，避免与调度算法双头漂移。

摘要：

- 仅考虑单一充电桩故障且该桩当前有车排队
- 故障发生后：停止计费 → 生成独立详单 → 暂停等候区叫号 → 按 `fault_dispatch_mode` 重调度
- 故障恢复后：统一按时间顺序重排

## 六、参数化功能（F-J01）

最终要求末注：`FastChargingPileNum / TrickleChargingPileNum / WaitingAreaSize / ChargingQueueLen` 在系统验收测试时可自由设置。

本系统采用"启动配置文件 + 管理员只读查看"方案：

- 参数由后端启动时从配置文件（或环境变量）加载
- 管理端仅提供 `GET /api/admin/system/config` 只读查看
- 运行时不允许修改；如需修改，需停机改配置后重启
- 系统验收批量测试时，场景参数通过 `/api/test/batch-simulate` 的 `scenario` 字段临时覆盖，不影响持久化配置

此方案避免了运行时修改参数可能引入的状态不一致风险，对应错误码 `1009`。

## 七、状态机

### 7.1 请求状态转移表

请求状态枚举：`WAITING_AREA`、`QUEUED`、`CHARGING`、`COMPLETED`、`COMPLETED_EARLY`、`CANCELLED`、`FAULT_INTERRUPTED`。

| 当前状态 | 可转移到 | 触发事件 |
|----------|----------|----------|
| （无） | `WAITING_AREA` | 用户创建请求 F-U02 |
| `WAITING_AREA` | `WAITING_AREA`（自环） | 等候区修改模式/电量 F-U04 / F-U05 |
| `WAITING_AREA` | `QUEUED` | 被叫号进入某桩队列 |
| `WAITING_AREA` | `CANCELLED` | 等候区取消 F-U06 |
| `QUEUED` | `CHARGING` | 上一辆车结束，队首开始充电 |
| `QUEUED` | `COMPLETED_EARLY` | 充电区取消 F-U07 |
| `QUEUED` | `FAULT_INTERRUPTED` | 所在桩故障，按策略重调度前已占位而取消该占位 |
| `CHARGING` | `COMPLETED` | 已充电量达到 `request_energy` |
| `CHARGING` | `COMPLETED_EARLY` | 用户提前结束 / 充电区取消 F-U07 |
| `CHARGING` | `FAULT_INTERRUPTED` | 所在桩故障 |
| `COMPLETED` | （终态） | —— |
| `COMPLETED_EARLY` | （终态） | —— |
| `CANCELLED` | （终态） | —— |
| `FAULT_INTERRUPTED` | （终态，作为本次详单的终态） | —— |

故障重调度会将车辆以**新请求**重新进入 `WAITING_AREA`，不是把原请求从 `FAULT_INTERRUPTED` 拉回。原请求的详单保留，下次进入队列是一条新请求。

### 7.2 充电桩状态转移表

桩状态枚举：`RUNNING`、`SHUTDOWN`、`FAULT`。

| 当前状态 | 可转移到 | 触发事件 |
|----------|----------|----------|
| （启动） | `RUNNING` 或 `SHUTDOWN`（取决于启动配置） | 系统启动 |
| `RUNNING` | `SHUTDOWN` | F-A01 关闭（需空闲且队列为空） |
| `RUNNING` | `FAULT` | F-A06 标记故障 |
| `SHUTDOWN` | `RUNNING` | F-A01 启动 |
| `FAULT` | `RUNNING` | F-A06 恢复故障 |

### 7.3 ASCII 状态机图

请求状态机：

```
                      +-----------------+
   create -->         |  WAITING_AREA   |
                      +-----------------+
                        |       |      \
                        v       v       v
                   +--------+  +-------+  +-----------+
                   | QUEUED |  | CANCELLED (终态)
                   +--------+  +-------+
                    |   |  \
                    v   v   \---> FAULT_INTERRUPTED (终态)
              +----------+   \
              | CHARGING |    \---> COMPLETED_EARLY (终态)
              +----------+
                |   |    \
                v   v     \---> FAULT_INTERRUPTED (终态)
         COMPLETED   COMPLETED_EARLY (终态)
         (终态)
```

桩状态机：

```
   +---------+        close (空闲且队列空)        +----------+
   | RUNNING | <----------------------------->    | SHUTDOWN |
   +---------+        start                       +----------+
        ^                                              ^
        | recover                                      |
        v                                              |
   +---------+                                         |
   |  FAULT  |   mark fault                            |
   +---------+                                         |
        ^-----------------------------------------------(不存在直接边)
```

### 7.4 调度触发事件总表

本表为 [调度模块输入输出约定.md](调度模块输入输出约定.md) §4 的镜像，便于本文档单文件自足。两处出现差异时以本表为最终用户可见口径。

| 模式 | 触发事件 |
