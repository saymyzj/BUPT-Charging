## 一、分工设计原则

老师已经确认本项目最终不再进行多组联合验收，因此分工需要从“跨组联调”切换为“本组系统完整交付”。

当前分工遵循以下原则：

1. 三人仍然通过少量冻结接口和冻结数据结构协作
2. 调度口径、测试输入口径和接口口径必须统一
3. 重点从“对外联调”转为“参数化验收场景、完整主流程、稳定展示和回归测试”
4. 不允许多人同时主改同一核心模块

## 三、冻结边界

为避免后期返工，项目开始后一周内必须冻结以下共享产物：

| 编号 | 冻结产物 | 负责人 | 说明 |
|------|----------|--------|------|
| FZ-01 | 状态枚举表 | 组长A | 请求状态、充电桩状态、支付状态等 |
| FZ-02 | 接口字段表 | 组长A + 成员B | 单次请求、批量测试、详单、账单、错误码 |
| FZ-03 | 验收场景输入结构 | 成员B + 组长A | 快慢充桩数量、等待区容量、队列解释方式 |
| FZ-04 | 调度模块输入输出约定 | 组长A | 调度引擎暴露给服务端的最小接口 |
| FZ-05 | 页面原型与展示口径 | 组长A + 成员C | 用户端/管理端页面结构与展示重点 |

冻结后禁止随意修改以下内容：

1. 接口路径
2. 请求字段名
3. 响应字段名
4. 状态枚举值
5. 详单字段
6. 账单字段
7. 批量测试 JSON 结构
8. 评测指标名称与含义

若必须修改，必须满足：

1. 提出修改人给出影响说明
2. 三人同步确认
3. 同步更新文档和实现

## 八、Git 协作规范、教程与操作流程

本项目使用 Git 做版本管理，远程仓库为：

`https://github.com/saymyzj/BUPT-Charging`

推荐分支模型：

- `main`：最终稳定版本、中期验收版本、最终答辩版本
- `develop`：日常集成分支
- `feature/a-algorithm-ui-review`：成员A长期工作分支
- `feature/b-backend-integration`：成员B长期工作分支
- `feature/c-frontend-testing`：成员C长期工作分支

### 8.1 协作原则

1. 不直接在 `main` 上开发
2. 高风险共用文件修改前先在组内说明
3. 合并到 `develop` 前先完成自测
4. 验收前由组长A统一从 `develop` 合并到 `main`
5. 每天开发前先同步 `develop`
6. 每次提交都写清楚提交信息

### 8.3 第一次接入仓库前要做什么

每位成员在开始开发前，先完成以下准备：

1. 接受 GitHub 仓库协作者邀请
2. 确认自己的 GitHub 账号拥有仓库访问权限
3. 本地安装 Git
4. 配置自己的 Git 用户名和邮箱

建议执行：

```bash
git config --global user.name "自己的姓名或GitHub用户名"
git config --global user.email "自己的GitHub邮箱"
```

### 8.4 推荐使用 SSH 连接 GitHub

建议统一使用 SSH，而不是 HTTPS。

先测试本机是否已配置好 SSH：

```bash
ssh -T git@github.com
```

如果看到类似下面的输出，说明 SSH 可用：

```text
Hi <github-username>! You've successfully authenticated...
```

如果还没有 SSH Key，可以执行：

```bash
ssh-keygen -t ed25519 -C "自己的GitHub邮箱"
cat ~/.ssh/id_ed25519.pub
```

把输出的公钥添加到 GitHub 的 SSH Keys 页面后，再执行一次：

```bash
ssh -T git@github.com
```

### 8.5 第一次克隆仓库的完整流程

推荐直接执行以下命令：

```bash
git clone git@github.com:saymyzj/BUPT-Charging.git
cd BUPT-Charging
git remote -v
git fetch origin
git branch -a
```

如果远程仓库还没有 `develop` 分支，由组长A执行：

```bash
git checkout -b develop
git push -u origin develop
```

### 8.7 每天开始开发前的标准流程

每天开始开发前，先把公共集成分支同步到本地，再同步到自己的工作分支。

标准命令如下：

```bash
git checkout develop
git pull origin develop
git checkout <自己的分支名>
git merge develop
git status
```

三位成员分别将 `<自己的分支名>` 替换为：

- `feature/a-algorithm-ui-review`
- `feature/b-backend-integration`
- `feature/c-frontend-testing`

```bash
git status
```

### 8.9 完成一轮开发后的标准提交流程

每次完成一个小功能或一轮稳定修改后，执行：

```bash
git status
git add .
git commit -m "feat: 完成xxx功能"
git push origin <自己的分支名>
```

更推荐按文件精确提交：

```bash
git status
git add docs/小组分工指南.md
git add backend/app.py
git commit -m "docs: 补充Git协作教程"
git push origin feature/a-algorithm-ui-review
```

### 8.10 提交信息规范

常用提交前缀：

- `feat:` 新功能
- `fix:` 修 bug
- `refactor:` 重构
- `docs:` 文档更新
- `test:` 测试相关
- `style:` 页面样式或 UI 优化

示例：

```bash
git commit -m "feat: 完成批量模拟接口"
git commit -m "fix: 修复过号后状态未更新"
git commit -m "docs: 更新分工与Git协作教程"
git commit -m "style: 优化排队状态页视觉层级"
```

### 8.11 需要继续拉取他人更新时怎么做

如果你已经在自己的分支上开发了一段时间，需要继续同步 `develop`，执行：

```bash
git checkout develop
git pull origin develop
git checkout <自己的分支名>
git merge develop
```

如果出现冲突：

1. 打开冲突文件手动修改
2. 解决后执行：

```bash
git add <冲突文件路径>
git commit -m "merge: 解决 develop 合并冲突"
```

### 8.12 合并到 develop 的流程

当某个功能开发完成并自测通过后，再合并到 `develop`。

建议流程：

1. 先在自己分支完成开发
2. 同步一次最新 `develop`
3. 自测通过后推送到远程
4. 由组长A或负责人统一合并到 `develop`

本地模拟合并前可以先执行：

```bash
git checkout develop
git pull origin develop
git checkout <自己的分支名>
git merge develop
git push origin <自己的分支名>
```

如果由组长A在本地完成合并，可执行：

```bash
git checkout develop
git pull origin develop
git merge <对方的分支名>
git push origin develop
```

### 8.13 合并到 main 的流程

只允许组长A把 `develop` 合并到 `main`。

在中期验收或最终验收冻结前执行：

```bash

git pull origin main
git checkout develop
git pull origin develop
git checkout main
git merge develop
git push origin main
```

### 8.14 中期和最终验收前的打标签流程

中期验收前，由组长A执行：


最终验收前，由组长A执行：

```bash

git push origin final-v1
```

### 8.15 常见撤回与恢复命令

撤回某个文件未提交的修改：

```bash
git restore <文件路径>
```

撤回已经 `add` 但还没提交的暂存：

```bash
git restore --staged <文件路径>
```

查看远程仓库信息：

```bash
git remote -v
```

### 8.16 不要做的事情

1. 不要直接在 `main` 上开发
2. 不要随意执行 `git push --force`
3. 不要长时间不同步 `develop`
4. 不要跨边界大改别人的模块
5. 不要在没沟通的情况下改高风险共用文档

## 九、成员之间允许依赖的内容

### 9.1 A -> B

成员B只依赖成员A提供的：

1. 调度模块输入输出约定
2. 状态机规则
3. 指标定义

### 9.2 A -> C

成员C只依赖成员A提供的：

1. 页面原型
2. UI 规范
3. 交互说明
4. 状态展示要求

### 9.3 B -> C

1. 接口字段
2. 统一响应结构
3. 错误码
4. 示例数据

### 9.4 禁止出现的高耦合情况

1. 三个人同时改同一个接口定义
2. 页面实现时临时改字段名
3. 服务端实现时临时改状态值
4. 测试时才首次对齐详单和账单结构
5. 最终验收前夕才开始验证批量测试输入

## 十一、结论

当前分工方案的核心思想是：

1. 组长A负责算法和统一口径
2. 成员B负责后端闭环和验收场景适配
3. 成员C负责前端展示和测试回归

由于不再进行联合验收，后期的工作重点应放在：

- 场景参数化
- 批量测试输入兼容
- 主流程稳定性
- 展示与答辩效果
