## 一、分工设计原则

本项目已进入开发中的需求调整重构阶段，不能再按“旧需求 + 局部修补”的方式协作。  
分工设计必须同时满足四个目标：

1. 严格对齐最终要求，不再保留旧“中央等待池”设计作为正式口径。
2. 不打散当前已经形成的三人协作模式和三增量框架。
3. 让每个核心模块都有明确所有权，避免多人同时主改同一高风险模块。
4. 把正式部署上线纳入职责和验收范围，而不是只要求本地可运行。

当前协作遵循以下原则：

1. 三人通过少量冻结接口、冻结字段和冻结状态语义协作。
2. 文档、代码、页面、验收输入必须统一口径，不允许双轨并存。
3. 高风险共用文档与核心接口修改前必须先在组内说明。
4. `develop` 是唯一公共集成基线，`main` 只保留里程碑和答辩稳定版本。
5. 任何成员都不应把“只在本地跑通”当作完成标准。

冻结后禁止随意修改以下内容：

1. 接口路径
2. 请求字段名
3. 响应字段名
4. 状态枚举值
5. 排队号语义
6. 详单与报表字段
7. 批量测试输入结构
8. 部署拓扑与访问方式

若必须修改，必须满足：

1. 提出修改的人给出影响说明。
2. 三人同步确认。
3. 同步更新文档。
4. 后续代码与页面一并跟进。

## 八、Git 协作规范、教程与操作流程

本项目使用 Git 进行版本管理。  
远程仓库以当前组内仓库为准，推荐统一使用 SSH。

推荐分支模型：

- `main`：稳定里程碑与最终答辩版本
- `develop`：公共集成分支
- `feature/a-algorithm-ui-review`：组长A长期工作分支
- `feature/b-backend-integration`：成员B长期工作分支
- `feature/c-frontend-testing`：成员C长期工作分支

### 8.1 协作原则

1. 不直接在 `main` 上开发
2. 每天开发前先同步 `develop`
3. 修改高风险共用文档前先组内说明
4. 合并到 `develop` 前先完成自测
5. 组长A统一负责把 `develop` 合并到 `main`
6. 除紧急修复外，不使用 `git push --force`

### 8.3 第一次接入仓库前要做什么

每位成员开始开发前，先完成：

1. 接受 GitHub 仓库协作者邀请
2. 确认本地已安装 Git
3. 配置 Git 用户名和邮箱
4. 配置 SSH Key

建议执行：

```bash
git config --global user.name "自己的姓名或GitHub用户名"
git config --global user.email "自己的GitHub邮箱"
```

### 8.4 推荐使用 SSH 连接 GitHub

先测试本机是否已配置 SSH：

```bash
ssh -T git@github.com
```

如果还没有 SSH Key：

```bash
ssh-keygen -t ed25519 -C "自己的GitHub邮箱"
cat ~/.ssh/id_ed25519.pub
```

将公钥添加到 GitHub 后再次执行：

```bash
ssh -T git@github.com
```

### 8.5 第一次克隆仓库的完整流程

```bash
git clone git@github.com:saymyzj/BUPT-Charging.git
cd BUPT-Charging
```

### 8.10 提交信息规范

常用前缀：

- `feat:` 新功能
- `fix:` 修复问题
- `refactor:` 重构
- `docs:` 文档更新
- `test:` 测试相关
- `style:` 页面与样式调整
- `merge:` 合并与冲突处理
- `wip:` 暂存当天进度

示例：

```bash
git commit -m "feat: 完成请求状态查询接口"
git commit -m "fix: 修复故障恢复后排序错误"
git commit -m "docs: 更新需求调整后的接口文档"
git commit -m "style: 优化管理员端监控页布局"
```

### 8.11 继续同步他人更新时怎么做

```bash
git checkout develop
git pull origin develop
git checkout <自己的分支名>
git merge develop
```

如果出现冲突：

1. 打开冲突文件手动解决
2. 解决后执行：

```bash
git add <冲突文件路径>
git commit -m "merge: 解决 develop 合并冲突"
```

### 8.12 合并到 `develop` 的流程

建议流程：

1. 在自己分支完成开发
2. 同步一次最新 `develop`
3. 自测通过后推送到远程
4. 由组长A或该模块负责人统一合并到 `develop`

组长A本地合并时可执行：

```bash
git checkout develop
git pull origin develop
git merge <对方的分支名>
git push origin develop
```

### 8.13 合并到 `main` 的流程

只允许组长A把 `develop` 合并到 `main`：

```bash
git checkout main
git pull origin main
git checkout develop
git pull origin develop
git checkout main
git merge develop
git push origin main
```

## 九、成员之间允许依赖的内容

### 9.1 A -> B

成员B只依赖组长A提供的：

1. 调度算法规则
2. 状态机规则
3. 故障与扩展模式边界
4. 验收输入解释口径

### 9.2 A -> C

成员C只依赖组长A提供的：

1. 页面展示口径
2. 状态文字与交互规则
3. 展示重点与答辩口径

### 9.3 B -> C

成员C只依赖成员B提供的：

1. 接口字段
2. 统一响应结构
3. 错误码
4. 示例数据

### 9.4 禁止出现的高耦合情况

1. 三个人同时改同一个接口定义
2. 页面实现时临时改字段名
3. 服务端实现时临时改状态值
4. 测试时才首次对齐详单和报表结构
5. 上线前夕才开始验证部署步骤

## 十一、结论

当前分工方案的核心思想是：

1. 组长A负责需求冻结、算法口径、部署拍板和最终收口。
2. 成员B负责后端闭环、状态机和数据输出。
3. 成员C负责前端展示、管理端和测试回归。

在需求调整已经明确、并且上线目标已经纳入范围的前提下，三人的共同目标不再是“先拼出能演示的本地版本”，而是：

- 严格按最终要求完成重构
- 保持协作节奏稳定
- 确保系统最终可以正式部署上线
