## 一、分工设计原则

本项目采用联合验收方式，且存在以下现实约束：
1. 为降低协作成本，各成员负责的内容必须尽量低耦合
因此，小组分工遵循以下原则：
1. 三人只通过少量冻结接口和冻结数据结构协作
2. 不允许多人同时主改同一核心模块

| FZ-02 | 联合验收接口字段表 | 组长A + 成员B | 单次模拟、批量模拟、详单、账单、错误码 |

| FZ-04 | 页面原型与交互说明 | 组长A | 用户端/管理端页面结构与视觉方向 |
| FZ-05 | 基础统计指标定义 | 组长A + 成员B | `avg_wait_seconds`、`avg_finish_seconds`、`station_utilization` 等 |

### 3.2 冻结后禁止随意修改的内容

以下内容冻结后，不得在未沟通的情况下擅自修改：

1. 接口路径
2. 请求字段名
3. 响应字段名
4. 状态枚举值
5. 详单字段
6. 账单字段
7. 批量模拟 JSON 结构
8. 评测指标名称与含义

若必须修改，必须满足：

1. 提出修改人给出影响说明
2. 三人同步确认
3. 同步更新文档和实现

## 八、Git 协作规范与教程

本项目使用 Git 做版本管理，远程仓库为：

`https://github.com/saymyzj/BUPT-Charging`

当前项目规模和协作方式最适合：

**`main` 稳定分支 + `develop` 集成分支 + 个人长期功能分支**

### 8.1 为什么采用这种 Git 方式

原因：

1. 三人并行开发，如果都直接改 `main`，冲突和误提交风险很高
2. 你们有接口冻结文档和联合验收规范，必须有一个专门的集成分支反复联调
3. 每人一个长期个人分支，比频繁新建很多短分支更适合课程项目

### 8.2 分支规划

| 分支名 | 用途 | 负责人 |
|--------|------|--------|
| `main` | 最终稳定版本、中期验收版本、最终答辩版本 | 组长A 维护 |
| `develop` | 日常集成分支，三人功能合并后的最新版本 | 组长A 主维护，成员B 协助 |
| `feature/a-algorithm-ui-review` | 组长A 的算法、规范、UI 原型与文档工作分支 | 组长A |
| `feature/b-backend-integration` | 成员B 的 Flask 后端、接口、批量模拟工作分支 | 成员B |
| `feature/c-frontend-testing` | 成员C 的 Vue 前端、管理端、测试与兼容性分支 | 成员C |
| `hotfix/*` | 紧急修复分支 | 由发现问题的人创建 |

### 8.3 第一次克隆与初始化

在开始克隆前，组员必须先完成：

1. 接受 GitHub 仓库协作者邀请
2. 确认自己的 GitHub 账号已具备仓库访问权限
3. 建议统一配置 SSH 方式访问 GitHub

推荐统一使用 SSH，而不是 HTTPS。

原因：

1. 后续会频繁 push / pull
2. SSH 配好后更稳定，不需要反复输入用户名和 token
3. 三人统一一种方式，后续排查问题更容易

#### 组员 SSH 接入检查

先测试：

```bash
ssh -T git@github.com
```

如果返回类似：

```bash
Hi <github-username>! You've successfully authenticated...
```

说明 SSH 已可用。

如果失败，则需要先生成 SSH key 并添加到 GitHub：

```bash
ssh-keygen -t ed25519 -C "自己的GitHub邮箱"
cat ~/.ssh/id_ed25519.pub
```

将输出的公钥添加到 GitHub 账户的 SSH Keys 中，再重新测试 `ssh -T git@github.com`。

```bash
git clone git@github.com:saymyzj/BUPT-Charging.git
cd BUPT-Charging
git fetch origin
```

如果远程仓库还没有 `develop`，由组长A执行：

```bash
git checkout -b develop
git push -u origin develop
```

然后三人分别创建自己的工作分支。

组长A：

```bash
git checkout develop
git pull origin develop
git checkout -b feature/a-algorithm-ui-review
git push -u origin feature/a-algorithm-ui-review
```

成员B：

```bash
git checkout develop
git pull origin develop
git checkout -b feature/b-backend-integration
git push -u origin feature/b-backend-integration
```

成员C：

```bash
git checkout develop
git pull origin develop
git checkout -b feature/c-frontend-testing
git push -u origin feature/c-frontend-testing
```

### 8.4 每天开发前的标准流程

```bash
git checkout develop
git pull origin develop
git checkout <自己的分支>
git merge develop
```

说明：

- 先同步 `develop`
- 再把 `develop` 合并到自己的分支
- 这样可以尽早发现冲突

### 8.4.1 组员第一次接入仓库的最简流程

成员B 和成员C 第一次接入仓库时，按下面顺序执行：

```bash
ssh -T git@github.com
git clone git@github.com:saymyzj/BUPT-Charging.git
cd BUPT-Charging
git fetch origin
git checkout develop
git pull origin develop
```

然后分别创建自己的分支。

### 8.5 提交代码的标准流程

```bash
git status
git add .
git commit -m "feat: 完成xxx功能"
git push origin <自己的分支>
```

提交信息规范：

- `feat:` 新功能
- `fix:` 修 bug
- `refactor:` 重构
- `docs:` 文档更新
- `test:` 测试相关
- `style:` 样式与 UI 优化

示例：

```bash
git commit -m "feat: 完成批量模拟接口"
git commit -m "fix: 修复过号后状态未更新"
git commit -m "docs: 更新冻结接口文档"
git commit -m "style: 优化用户端排队状态页"
```

### 8.6 合并规则

#### 合并到 `develop`

适用场景：

- 新功能完成
- bug 修复完成
- 文档更新完成

规则：

1. 先在自己分支完成自测
2. 再发起合并到 `develop`
3. 至少另一位成员快速 review
4. 合并后本地重新拉取最新 `develop`

#### 合并到 `main`

适用场景：

- 中期验收冻结版本
- 最终验收冻结版本

规则：

1. 必须先通过 `develop` 上的联调和回归测试
2. 只允许组长A从 `develop` 合并到 `main`
3. 不允许个人工作分支直接合并到 `main`

### 8.7 不同成员与 Git 的边界

#### 组长A

- 主要改：
  - 算法相关
  - 规范文档
  - 原型与静态页面
- 不直接大改成员B的后端实现文件
- 不直接大改成员C的页面实现文件

#### 成员B

- 主要改：
  - Flask 后端
  - SQLite 数据层
  - 联合验收接口
  - 批量模拟
- 不直接改成员C负责的 Vue 页面
- 不直接改成员A已冻结的算法口径

#### 成员C

- 主要改：
  - Vue 前端
  - 管理端
  - 测试脚本
  - 兼容性说明
- 不直接改成员B负责的接口语义
- 不直接改成员A负责的算法规则

### 8.8 高风险共用文件修改规则

以下文件属于高风险共用文件，修改前必须先在组内说明：

1. `docs/冻结接口文档.md`
2. `docs/功能设计.md`
3. `docs/架构设计.md`
4. `调度算法.md`
5. `联合验收统一接口与数据结构规范说明.md`

修改流程：

1. 先在群里说明要改什么
2. 由组长A确认是否允许修改
3. 修改后必须同步更新相关实现

## 九、低耦合协作约定

### 8.1 成员之间允许依赖的内容

#### A -> B

成员B只依赖成员A提供的：

1. 调度模块输入输出约定
2. 状态机规则
3. 指标定义
4. 联合验收对外口径

#### A -> C

成员C只依赖成员A提供的：

1. 页面原型
2. UI 规范
3. 交互说明
4. 状态展示要求

#### B -> C

成员C只依赖成员B提供的：

1. 接口字段
2. 统一响应结构
3. 错误码
4. 示例数据

### 8.2 禁止出现的高耦合情况

1. 三个人同时改同一个接口定义
2. 页面实现时临时改字段名
3. 服务端实现时临时改状态值
4. 测试时才首次对齐详单/账单结构
5. 联合验收前夕才开始跨设备通信验证

## 十、最终报告撰写分工

最终报告应由三人分工撰写。

## 10.1 报告章节建议划分

| 报告章节 | 主责人 | 配合人 |
|----------|--------|--------|
| 项目背景与需求分析 | 组长A | 全员 |
| 功能设计 | 成员C | 组长A |
| 架构设计 | 成员B | 成员C |
| 调度算法设计 | 组长A | 成员B |
| 数据结构与接口设计 | 成员B | 组长A |
| 用户端与管理端页面设计 | 成员C | 组长A |
| 联合验收规范与批量模拟设计 | 成员B | 组长A |
| 测试方案与测试结果 | 成员C | 成员B |
| 展示效果与项目总结 | 组长A | 成员C |

## 10.2 报告合稿流程

1. 各自先写自己的负责章节
2. 组长A统一术语、口径、图表和章节衔接
3. 成员B统一接口与数据结构描述
4. 成员C统一页面截图、图表和展示部分
5. 最终由组长A完成全稿终审

## 十一、最终结论

本分工方案的核心思想是：

- 组长A承担高决策、高对外沟通、高展示价值的工作
- 成员B承担高工程量、高联调量的工作
- 成员C承担高页面实现量、高测试量、高兼容性验证量的工作
