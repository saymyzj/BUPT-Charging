# 调度算法

> 说明：本文件整理自确认文档中的调度算法、异常处理、状态机与数据结构部分，用于单独描述系统调度方案。

## 一、核心调度算法设计

### 1.1 为什么内部实现不采用固定充电桩等待队列

在往届学长学姐的报告中，常见的设计是每个充电桩维护一个固定的等待队列，用户一提交请求就被分配到某个桩队列中。这种设计虽然实现简单，但在实际调度过程中会引出很多问题：

- 用户取消后，原桩队列需要局部重排
- 用户过号后，原桩是否继续空等不明确
- 桩故障后，原队列用户需要整体迁移
- 某桩空闲而其他桩繁忙时，容易出现资源浪费

因此，本系统在内部实现上采用中央等待池和动态分配，而不是静态桩队列。

### 1.2 核心思想

系统按充电类型建立两个中央等待池：

- `FAST_POOL`
- `SLOW_POOL`

用户提交请求后：

- 先进入对应类型的等待池
- 系统只展示预计等待时间，不立即锁定具体桩号
- 当某个同类型桩即将可用时，再根据当前状态动态分配最终桩号

### 1.3 验收输入兼容策略

老师确认最终验收中，“充电桩排队队列长度”可能有两种解释，因此系统需要做输入兼容，而不强迫验收方理解内部建模细节。

兼容方式如下：

1. 若验收输入给出统一的“队列长度”参数，则将其解释为每个充电桩的排队容量上限
2. 若验收输入给出“A 桩正在充电，有 3 人排队，B 桩空闲，C 桩正在充电，有 2 人排队”这类描述，则将其解释为桩级初始状态快照
3. 上述两种输入都会先被转换为：
   - 充电桩初始状态 `status(s)`
   - 各桩的初始最早可用时间 `available_time(s)`
   - 初始待服务请求集合
4. 转换完成后，再交给统一的中央等待池调度器执行预测和重调度

### 1.4 调度目标

调度策略采用“带老化补偿的短作业优先”思想，综合目标如下：

1. 优先让单个用户本次充电总时长最短，即 `等待时间 + 充电时间` 最短
2. 在扩展目标下，尽量让多台车辆总完成时长最短
3. 保证长作业不会长期饥饿
4. 在异常后尽快恢复系统吞吐

### 1.5 基本变量定义

设用户请求为 `u`，充电桩为 `s`，当前时刻为 `now`。

| 符号 | 含义 |
|------|------|

| `submit_time(u)` | 用户提交请求时间 |
| `wait_time(u)` | 当前已等待时长 |
| `retry_count(u)` | 用户被过号或重入队次数 |
| `fault_requeue(u)` | 是否因故障重新入队 |
| `remaining_energy(u)` | 剩余待充电量 |
| `actual_energy(u)` | 当前已实际充入电量 |
| `battery_limit(u)` | 用户车辆实际还能充入的电量上限，模拟环境中由系统维护 |

#### 1.5.2 充电桩侧变量

| 符号 | 含义 |
|------|------|
| `type(s)` | 充电桩类型，快充或慢充 |
| `power(s)` | 充电桩额定功率 |
| `status(s)` | 当前状态 |
| `available_time(s)` | 最早可接收下一用户的时间 |

### 1.6 充电时间估算公式

对于任一请求 `u`，其预计服务时长记为：

`service_time(u, s) = remaining_energy(u) / power(s)`

若同类型充电桩功率相同，也可简化为：

`service_time(u) = remaining_energy(u) / power(mode(u))`

说明：

- 调度阶段默认 `remaining_energy(u) = request_energy(u)`
- 单位可统一为分钟或秒
- 若需要更真实，可加入固定接入时间常数 `setup_time`
- 若用户实际可充入电量小于申报值，则该预测值会在充电过程中被动态修正

扩展形式：

`service_time(u, s) = setup_time + remaining_energy(u) / power(s)`

实际执行时，本次会话的真实充电量定义为：

`actual_charge_energy(u) = min(request_energy(u), battery_limit(u))`

因此真实充电时长可记为：

`actual_service_time(u, s) = setup_time + actual_charge_energy(u) / power(s)`

若 `actual_charge_energy(u) < request_energy(u)`，则说明用户提前充满，系统应提前释放充电桩并触发重调度。

### 1.7 充电桩最早可用时间

充电桩 `s` 的最早可用时间定义为：

`available_time(s) = charge_end_time(s) + leave_buffer`

其中：

- `charge_end_time(s)` 为当前会话预计结束时间
- `leave_buffer` 为用户充满后允许挪车的缓冲时间

若当前桩状态为空闲，则：

`available_time(s) = now`

若用户超时未挪车，则实际可用时间继续后移。

### 1.8 预计等待时间与预计完成时间

若将用户 `u` 分配给桩 `s`，则其预测等待时间为：

`pred_wait(u, s) = max(0, available_time(s) - now)`

预测完成时间为：

`pred_finish(u, s) = pred_wait(u, s) + service_time(u, s)`

系统对同类型所有可用桩进行比较，优先选择 `pred_finish(u, s)` 最小的候选桩。

### 1.9 用户动态优先级公式

为避免单纯 FCFS 效率低，也避免纯短作业优先导致长作业饥饿，系统采用带老化补偿的动态优先级：

`priority(u) = a * wait_time(u) + b * retry_bonus(u) + c * fault_bonus(u) - d * service_time(u) - e * no_show_penalty(u)`

其中：

- `wait_time(u)`：已等待越久，优先级越高
- `retry_bonus(u)`：过号后重新入队时给予部分保留或补偿
- `fault_bonus(u)`：因桩故障被迫中断时，提高优先级
- `service_time(u)`：预计服务时间越短，越倾向优先安排
- `no_show_penalty(u)`：叫号未到场时可施加惩罚

说明：

- `a、b、c、d、e` 为可配置权重
- 若课程实现需简化，可先使用 `priority = wait_time - k * service_time`

### 1.10 调度算法主流程

#### 1.10.1 新请求到达

1. 用户提交 `charge_mode` 和 `request_energy`
2. 系统生成请求记录 `charge_request`
3. 按 `charge_mode` 将请求放入对应中央等待池
4. 调度器触发一次预测排程
5. 返回当前预计等待时长和排队状态

说明：

- 此时系统只依据 `request_energy` 进行预测，不保证预测值与最终实际值完全一致
- 若实际充电过程提前结束，系统会重新修正后续预测结果

#### 1.10.2 预测排程

对某一类型的等待池执行如下步骤：

1. 取出该类型所有可用或未来可用的桩
2. 计算各桩 `available_time`
3. 计算等待池中每个请求的 `priority`
4. 按优先级排序，若优先级相同则按提交时间排序
5. 以“最早可用桩优先”的方式滚动模拟分配
6. 得到每个请求的：
   - 预计等待时间
   - 预计开始时间
   - 预计完成时间
7. 将上述预测结果写回请求记录，用于前端展示

#### 1.10.3 临近确认与最终分配

当某个桩 `s` 距离可用时刻小于阈值 `T_call` 时：

1. 从对应等待池中选择当前最适合的用户 `u`
2. 向用户发送叫号通知
3. 用户需在 `confirm_timeout` 内确认
4. 确认后，系统将该桩临时保留 `arrival_timeout`
5. 用户在规定时间内完成接入，则开始充电
6. 若未确认或未到场，则判为过号，系统重新选择下一个用户
7. 若充电过程中车辆提前充满，则立即结束会话，释放充电桩并重调度

### 1.11 事件驱动重调度

调度器并不是只在新请求到达时运行，而是在关键事件发生时统一触发重调度。

#### 1.11.1 触发事件

以下事件均会触发调度器重新计算：

1. 新用户提交请求
2. 用户取消排队
3. 用户确认叫号
4. 用户过号
5. 用户开始充电
6. 用户中断充电
7. 充电完成
8. 用户超时未挪车
9. 充电桩故障
10. 故障恢复
11. 用户提前充满

#### 1.11.2 重调度动作

每次重调度时，系统执行：

1. 更新等待池状态
2. 更新充电桩状态与 `available_time`
3. 更新请求优先级
4. 重新计算预测等待时长
5. 若有临近空闲桩，则重新选择叫号用户

## 二、异常场景与状态处理规则

### 2.1 用户取消排队

规则：

- 若尚未开始充电，允许取消
- 系统将请求状态更新为 `CANCELLED`
- 从等待池移除
- 触发重调度

### 2.2 用户中断充电

规则：

- 按已充电量结算当前费用
- 若视为充电结束，则释放充电桩
- 若允许续充，则将剩余电量作为新请求重新入池
- 对续充用户可赋予一定补偿优先级

### 2.3 用户申报量与实际量不一致

该场景视为系统常规场景，而非异常输入。

规则：

- 用户提交的 `request_energy` 用于预测排队和预计等待时长
- 充电过程中实时记录 `actual_energy`
- 若实际车辆可充入电量小于用户申报量，则在达到车辆上限时提前结束
- 本次费用按 `actual_energy` 和实际充电时长结算
- 充电桩提前释放后，系统立即触发重调度，更新后续用户预计等待时间

建议课程实现采用以下统一规则：

`最终结算量 = min(用户申报量, 车辆实际可充入量)`

### 2.4 过号处理

规则：

- 用户收到叫号后，需在限定时间内确认并到场
- 超时未确认或未接入，状态记为 `NO_SHOW`
- 用户可重新进入等待池
- 重新入池时可采用以下之一：
  - 重新排到末尾
  - 保留部分等待权重
  - 增加一定惩罚项

建议课程实现采用：

`重新入池 + 轻度惩罚`

### 2.5 充满后未及时挪车

规则：

- 充电完成后，状态进入 `WAITING_TO_LEAVE`
- 允许缓冲时间 `leave_buffer`
- 若超时，可增加占位费
- 在车辆未挪走前，该桩不可重新分配

### 2.6 充电桩故障

规则：

- 若故障发生在空闲态：直接标记 `FAULT`，不参与调度
- 若故障发生在充电中：
  - 记录已充电量
  - 将剩余电量生成新请求重新入池
  - 增加 `fault_bonus`
- 若故障恢复：恢复为 `IDLE`，重新参与调度

## 三、核心状态机设计

### 3.1 请求状态

建议请求状态如下：

| 状态 | 含义 |
|------|------|
| `PENDING` | 已提交，尚未进入预测 |
| `WAITING` | 在中央等待池中等待 |
| `CALLED` | 已被叫号，等待用户确认 |
| `CONFIRMED` | 用户已确认，等待接入 |
| `CHARGING` | 正在充电 |
| `INTERRUPTED` | 中断充电 |
| `COMPLETED` | 充电完成 |
| `COMPLETED_EARLY` | 提前充满并结束 |
| `CANCELLED` | 用户取消 |
| `NO_SHOW` | 过号未到 |
| `FAULT_REQUEUE` | 因故障重新入队 |

### 3.2 充电桩状态

建议充电桩状态如下：

| 状态 | 含义 |
|------|------|
| `IDLE` | 空闲 |
| `RESERVED` | 已叫号并临时保留 |
| `CHARGING` | 正在充电 |
| `WAITING_TO_LEAVE` | 充满待挪车 |
| `FAULT` | 故障，不可用 |

### 4.2 运行期数据结构

为提高调度效率，可在内存中维护以下结构：

#### 4.2.1 等待池优先队列

- `fast_pool_heap`
- `slow_pool_heap`

作用：

- 按 `priority_score` 维护等待请求
- 堆顶为当前最优先调度的请求

#### 4.2.2 充电桩最早可用堆

- `fast_station_heap`
- `slow_station_heap`

作用：

- 按 `available_time` 排序
- 堆顶为最早空闲的同类型充电桩

#### 4.2.3 事件队列

- `event_queue`

作用：

- 接收“新请求、取消、过号、故障”等事件
- 触发调度器增量更新或全量重排
