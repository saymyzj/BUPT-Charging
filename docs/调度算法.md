# 智能充电桩调度计费系统：调度算法

> 版本：V3.0
> 状态：最终冻结
> 冻结日期：2026-04-19
> 维护人：组长A
> 目标：严格按最终要求定义普通调度、故障重调度与扩展调度

## 一、基础定义

### 1.1 输入参数

- `FastChargingPileNum`
- `TrickleChargingPileNum`
- `WaitingAreaSize`
- `ChargingQueueLen`
- `dispatch_mode`
- `fault_dispatch_mode`

四个基础参数由启动配置文件加载，运行时不可改，详见 [功能设计.md](功能设计.md) §6。

### 1.2 队列语义

- 等候区容量为 `WaitingAreaSize`
- 每个充电桩队列容量为 `ChargingQueueLen`
- `ChargingQueueLen` 包含队首充电位

### 1.3 排队号

- 快充：`F1, F2, ...`
- 慢充：`T1, T2, ...`
- 同模式编号运行期间单调递增
- 系统或场景重置后从 `1` 重新编号

### 1.4 符号约定

| 符号 | 含义 |
|------|------|
| `u` | 待调度车辆（对应一个 `ChargeRequest`） |
| `s` | 候选充电桩 |
| `power(s)` | 桩 `s` 的额定功率（kW），快充=30，慢充=10 |
| `request_energy(u)` | 车辆 `u` 请求充电度数（kWh） |
| `self_charge_time(u, s)` | 车辆 `u` 在桩 `s` 上充电所需时间（秒）= `request_energy(u) / power(s) × 3600` |
| `wait_time(u, s)` | 车辆 `u` 在桩 `s` 队列中的等待时间（秒） |
| `finish_time(u, s)` | 车辆 `u` 完成时间（秒）= `wait_time(u, s) + self_charge_time(u, s)` |

## 二、普通调度 `NORMAL`

### 2.1 触发条件

当任意充电桩队列出现空位时触发。

### 2.2 规则

1. 仅在对应模式内调度
2. 取该模式等候区排队号最靠前的车辆
3. 在所有同模式且存在空位的桩队列中，计算该车加入后完成充电所需总时长
4. 将该车加入总时长最短的桩队列

### 2.3 计算公式

对候选车辆 `u` 加入候选桩 `s`：

- `wait_time(u, s) = station_queue_remaining_time_sum(s)`
- `self_charge_time(u, s) = request_energy(u) / power(s) × 3600`（秒）
- `finish_time(u, s) = wait_time(u, s) + self_charge_time(u, s)`

选择 `finish_time(u, s)` 最小的桩。

`station_queue_remaining_time_sum(s)` 的严格定义：

```
station_queue_remaining_time_sum(s) = Σ_i ( remaining_energy_i / power(s) × 3600 )
  其中 i 遍历桩 s 当前队列中的所有车辆
  对队首正在充电的车辆，remaining_energy_i = request_energy_i - 已累计充电量
  对队列中其他车辆，remaining_energy_i = request_energy_i
```

### 2.4 说明

普通模式下：

- 等候区取车顺序严格遵守排队号先后
- 只在"进入哪个同模式桩队列"这一步做最优选择

## 三、故障处理

> 本节是故障算法的唯一来源；面向用户/管理端的交互语义见 [功能设计.md](功能设计.md) §5 和 §7，验收场景见 [故障与扩展场景指南.md](故障与扩展场景指南.md)。

### 3.1 故障发生的统一前置动作

仅考虑：

- 单一充电桩故障
- 且故障桩当前有车排队

发生故障后，统一先执行：

1. 故障桩状态设为 `FAULT`
2. 当前正在充电车辆停止计费
3. 当前已充电部分生成一条独立详单，状态 `FAULT_INTERRUPTED`
4. 暂停等候区叫号服务

### 3.2 故障策略 `PRIORITY`

定义：

- "故障队列"指故障桩上尚未完成服务的排队车辆集合（不含已生成 `FAULT_INTERRUPTED` 详单的那辆车，那辆车以新请求重新入等候区）

规则：

1. 保持其它同类型桩继续正常出队
2. 当其它同类型桩出现空位时，优先从故障队列补位
3. 故障队列未清空前，不从等候区继续叫号
4. 故障队列全部调度完成后，恢复等候区叫号

### 3.3 故障策略 `TIME_ORDER`

定义：

- 参与重排的集合由两部分构成：
  - 故障桩队列中的未开始充电车辆
  - 其它同类型桩中尚未开始充电的车辆

规则：

1. 将上述车辆按排队号码统一排序
2. 依排序结果重新装配到所有可用同类型桩队列
3. 重排完成前，暂停等候区叫号
4. 重排完成后，恢复等候区叫号

### 3.4 故障恢复

故障桩恢复时：

1. 若其它同类型桩中仍存在未开始充电车辆
2. 暂停等候区叫号
3. 将这些车辆合并，按排队号码先后顺序重新调度
4. 调度完成后恢复等候区叫号

故障恢复不再分 `PRIORITY` 与 `TIME_ORDER`，统一执行时间顺序重排。

## 四、扩展调度

### 4.1 单次联合调度 `EXT_SINGLE_BATCH`

#### 4.1.1 触发条件

- 同时出现多个充电区空位

#### 4.1.2 规则

1. 本次联合调度中，不考虑同模式排队先后
2. 仍保持快充车只进快充桩、慢充车只进慢充桩
3. 选出一组待进入充电区的车辆及其分配方案
4. 目标是该批车辆累计等待时间与累计充电时间之和最小

#### 4.1.3 目标函数

```
min  Σ_{j ∈ 本次联合调度车辆} ( wait_time(u_j, s_j) + self_charge_time(u_j, s_j) )
s.t. charge_mode(u_j) == charge_mode(s_j)   对所有 j
     每个 s_j 同一时刻占用不超过其队列容量
```

#### 4.1.4 互斥边界

- 不考虑请求修改
- 不考虑故障
- 与普通模式互斥

### 4.2 批量调度 `EXT_FULL_BATCH`

#### 4.2.1 触发条件

- 到达车辆数等于全站总车位数（充电区 + 等候区）

#### 4.2.2 规则

1. 一次批量调度中不区分快充和慢充
2. 不考虑车辆到达先后顺序
3. 所有车辆均可分配到任意类型充电桩
4. 目标是一整批车辆累计等待时间与累计充电时间之和最小
5. 一批结束后再进入下一批

#### 4.2.3 目标函数

```
min  Σ_{j ∈ 整批车辆} ( wait_time(u_j, s_j) + self_charge_time(u_j, s_j) )
s.t. 每个 s_j 可以是快充或慢充任意一种
     每个 s_j 同一时刻占用不超过其队列容量
```

#### 4.2.4 互斥边界

- 不考虑请求修改
- 不考虑故障
- 与普通模式互斥

## 五、调度模式切换规则

### 5.1 系统只允许一种正式调度模式处于生效状态

可选：

- `NORMAL`
- `EXT_SINGLE_BATCH`
- `EXT_FULL_BATCH`

### 5.2 模式切换要求

- 由管理员运行时切换（通过 `PUT /api/admin/system/dispatch-mode`）
- 切换动作必须记录到系统事件日志
- 验收和批量测试时应显式写明当前模式

## 六、需要始终保持一致的口径

以下口径在所有文档和后续实现中必须一致：

1. 正式运行模型是"等候区 + 每桩固定队列"
2. 普通模式下先取等候区头车，再选最优同模式桩
3. 故障发生后暂停等候区叫号
4. 故障恢复固定按时间顺序重排
5. 扩展模式下不考虑请求修改和故障
6. 四个基础参数只能启动时配置，运行时不可改
