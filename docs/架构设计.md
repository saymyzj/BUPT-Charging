## 一、架构总览

### 1.1 架构风格

系统采用前后端分离的 B/S 架构：

- **前端**：Vue 单页应用（SPA），负责用户交互和数据展示
- **后端**：Python + Flask RESTful API 服务，负责业务逻辑、调度计算和数据持久化
- **通信**：HTTP + JSON，支持局域网跨设备访问

### 1.2 技术栈确定

结合团队现状与联合验收要求，技术栈最终确定如下：

| 层次 | 技术选择 | 说明 |
|------|----------|------|
| 前端 | Vue | 负责用户端和管理端页面实现 |
| 后端 | Python + Flask | 负责 RESTful API、调度逻辑、批量模拟、联合验收接口 |
| 数据库 | SQLite | 开发和演示阶段使用，轻量、易携带、便于联合验收环境快速启动 |
| 通信协议 | HTTP + JSON | 满足跨设备和跨平台联调要求 |
| 版本管理 | Git + GitHub | 远程仓库为 `https://github.com/saymyzj/BUPT-Charging` |

技术选型原则：

1. 以快速落地和联合验收稳定性为优先
2. 降低环境配置成本，方便 mac 和 Windows 运行
3. 便于与 AI coding 配合，提高开发效率

### 1.4 核心设计决策

| 决策项 | 选择 | 原因 |
|--------|------|------|
| 调度模型 | 中央等待池 + 动态分配 | 灵活处理取消、过号、故障等动态事件 |
| 桩队列模型 | 无固定桩队列 | 桩只维护自身状态，不绑定用户队列 |
| 调度触发方式 | 事件驱动 | 在关键事件发生时统一重调度 |
| 通信协议 | HTTP + JSON | 满足联合验收跨平台要求 |
| 前后端关系 | Vue + Flask 前后端分离 | 便于多端接入和联合验收 |
| 后端语言 | Python | 开发效率高，适合快速实现调度与接口 |
| 数据库 | SQLite | 便于演示部署和跨平台迁移 |

#### 2.2.3 等待池管理模块

| 职责 | 说明 |
|------|------|
| 维护快充池 | `FAST_POOL` 优先队列 |
| 维护慢充池 | `SLOW_POOL` 优先队列 |
| 入池 | 新请求或故障重入队请求加入池 |
| 出池 | 叫号分配时从池中取出 |
| 移除 | 用户取消时从池中删除 |
| 优先级更新 | 重调度时重新计算所有请求优先级 |

**内部接口**（供调度引擎调用）：

- `add_to_pool(request, pool_type)`
- `remove_from_pool(request_id, pool_type)`
- `peek_top(pool_type)` — 获取最高优先级请求
- `pop_top(pool_type)` — 取出最高优先级请求
- `update_priorities(pool_type)` — 重算所有优先级
- `get_pool_size(pool_type)` — 获取池中请求数量
- `get_all_requests(pool_type)` — 获取池中所有请求（用于预测排程）

#### 2.2.4 调度引擎模块

本系统的核心模块，负责动态调度决策。

| 职责 | 说明 |
|------|------|
| 优先级计算 | 按公式计算每个请求的 `priority_score` |
| 预测排程 | 模拟分配，计算预计等待时间和完成时间 |
| 叫号决策 | 桩即将可用时选择最合适的用户 |
| 重调度 | 事件触发后更新全局预测 |

**调度算法核心**：

优先级公式：

```
priority(u) = a * wait_time(u) + b * retry_bonus(u) + c * fault_bonus(u)
              - d * service_time(u) - e * no_show_penalty(u)
```

预测排程流程：

1. 取出该类型所有可用或未来可用的桩
2. 计算各桩 `available_time`
3. 计算等待池中每个请求的 `priority`
4. 按优先级排序（相同优先级按提交时间排序）
5. 以最早可用桩优先的方式滚动模拟分配
6. 得到每个请求的预计等待时间、预计开始时间、预计完成时间
7. 写回请求记录

**事件驱动接口**：

- `on_new_request(request)` — 新请求到达
- `on_cancel(request_id)` — 用户取消
- `on_confirm(request_id)` — 用户确认叫号
- `on_no_show(request_id)` — 用户过号
- `on_charge_start(request_id, station_id)` — 开始充电
- `on_charge_end(request_id, station_id)` — 充电完成
- `on_charge_interrupt(request_id, station_id)` — 中断充电
- `on_early_complete(request_id, station_id)` — 提前充满
- `on_leave_timeout(request_id, station_id)` — 超时未挪车
- `on_station_fault(station_id)` — 桩故障
- `on_station_recover(station_id)` — 桩恢复

每个事件处理器内部执行：

1. 更新等待池状态
2. 更新充电桩状态与 `available_time`
3. 更新请求优先级
4. 重新计算预测等待时长
5. 若有临近空闲桩，重新选择叫号用户

#### 2.2.5 充电桩状态模块

| 职责 | 说明 |
|------|------|
| 桩状态维护 | 维护每个桩的实时状态 |
| 可用时间计算 | 计算 `available_time` |
| 桩可用堆维护 | 按 `available_time` 排序的优先队列 |
| 故障管理 | 标记故障和恢复 |

**内部数据结构**：

- `fast_station_heap` — 快充桩按 `available_time` 排序
- `slow_station_heap` — 慢充桩按 `available_time` 排序

**关键计算**：

```
available_time(s) = charge_end_time(s) + leave_buffer    (充电中)
available_time(s) = now                                   (空闲)
```

#### 2.2.6 充电会话模块

| 职责 | 说明 |
|------|------|
| 创建会话 | 用户开始充电时创建 `charging_session` |
| 实时更新 | 更新 `actual_energy`，检测是否提前充满 |
| 结束会话 | 正常完成、提前充满、中断或故障时结束 |
| 挪车管理 | 管理 `WAITING_TO_LEAVE` 状态和超时检测 |

**充电过程模拟**：

- 实际充电量 = `min(request_energy, battery_limit_energy)`
- 充电时长 = `actual_charge_energy / power(s)`
- 若 `actual_energy` 达到 `battery_limit_energy`，触发提前充满

#### 2.2.7 计费与账单模块

| 职责 | 说明 |
|------|------|
| 费用计算 | 根据计费规则计算各项费用 |
| 详单生成 | 生成符合联合验收规范的统一详单 |
| 账单生成 | 生成符合联合验收规范的统一账单 |
| 支付处理 | 模拟支付流程，更新支付状态 |

**计费公式**：

```
energy_fee = actual_energy * energy_price
time_fee = actual_service_minutes * time_price
occupancy_fee = max(0, overtime_minutes) * occupancy_price
total_fee = energy_fee + time_fee + occupancy_fee
```

#### 2.2.8 通知服务模块

| 职责 | 说明 |
|------|------|
| 叫号提醒 | 请求状态变为 `CALLED` 时通知用户 |
| 充满提醒 | 充电完成或提前充满时通知 |
| 挪车提醒 | 进入 `WAITING_TO_LEAVE` 时通知 |
| 过号提醒 | 状态变为 `NO_SHOW` 时通知 |
| 异常提醒 | 故障发生时通知受影响用户 |

**实现方式**：

- 基础方案：前端轮询通知接口
- 增强方案：WebSocket 推送

**通知接口**：

| 接口 | 方法 | 路径 |
|------|------|------|
| 获取通知列表 | GET | `/api/notifications` |
| 标记已读 | POST | `/api/notifications/{id}/read` |

#### 2.2.9 批量模拟模块

| 职责 | 说明 |
|------|------|
| 接收测试用例 | 解析批量测试 JSON |
| 事件回放 | 按时间顺序模拟所有事件 |
| 结果汇总 | 汇总每个用户的详单、账单和统计指标 |

**处理流程**：

1. 接收批量测试用例 JSON
2. 初始化调度系统状态（清空等待池、重置桩状态）
3. 将所有用户事件按 `request_time` 排序
4. 依次处理每个事件，按统一优先级解决同时刻冲突
5. 模拟完成后，收集每个用户的详单和账单
6. 计算汇总统计指标
7. 返回统一格式输出

#### 2.2.10 统计分析模块

| 职责 | 说明 |
|------|------|
| 指标计算 | 计算平均等待时间、完成时间、利用率等 |
| 数据聚合 | 按时间段、类型等维度汇总 |
| 报表导出 | 输出 JSON 或 CSV 格式 |

**评测指标**（遵循联合验收规范）：

| 指标 | 计算方式 |
|------|----------|
| 平均等待时间 | `sum(wait_time) / count(users)` |
| 平均完成时间 | `sum(wait_time + charge_time) / count(users)` |
| 最大等待时间 | `max(wait_time)` |
| 充电桩利用率 | `sum(busy_time) / sum(total_available_time)` |
| 吞吐量 | `completed_users / total_time` |

## 三、数据层设计

### 3.1 持久化数据库

采用 SQLite 存储业务数据。考虑到本项目为课程设计与联合验收场景，SQLite 更适合快速启动、跨平台迁移和演示环境部署。

#### 3.1.1 用户表 `user`

```sql
CREATE TABLE user (
    id              BIGINT PRIMARY KEY AUTO_INCREMENT,
    username        VARCHAR(64) NOT NULL UNIQUE,
    password_hash   VARCHAR(256) NOT NULL,
    role            VARCHAR(16) NOT NULL DEFAULT 'USER',  -- USER / ADMIN
    balance         DECIMAL(10,2) NOT NULL DEFAULT 0.00,
    created_at      DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```

#### 3.1.2 充电请求表 `charge_request`

```sql
CREATE TABLE charge_request (
    id                      BIGINT PRIMARY KEY AUTO_INCREMENT,
    request_id              VARCHAR(32) NOT NULL UNIQUE,       -- 业务编号如 REQ001
    user_id                 BIGINT NOT NULL,
    charge_mode             VARCHAR(8) NOT NULL,               -- FAST / SLOW
    request_energy          DECIMAL(10,2) NOT NULL,
    remaining_energy        DECIMAL(10,2) NOT NULL,
    actual_energy           DECIMAL(10,2) NOT NULL DEFAULT 0,
    battery_limit_energy    DECIMAL(10,2),
    status                  VARCHAR(20) NOT NULL DEFAULT 'PENDING',
    submit_time             DATETIME NOT NULL,
    estimated_wait_seconds  INT,
    estimated_start_time    DATETIME,
    estimated_finish_time   DATETIME,
    estimated_service_seconds INT,
    actual_service_seconds  INT,
    priority_score          DECIMAL(10,4) DEFAULT 0,
    retry_count             INT NOT NULL DEFAULT 0,
    no_show_count           INT NOT NULL DEFAULT 0,
    fault_requeue_flag      BOOLEAN NOT NULL DEFAULT FALSE,
    assigned_station_id     BIGINT,
    last_called_at          DATETIME,
    confirmed_at            DATETIME,
    created_at              DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at              DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES user(id),
    FOREIGN KEY (assigned_station_id) REFERENCES charging_station(id)
);
```

#### 3.1.3 充电桩表 `charging_station`

```sql
CREATE TABLE charging_station (
    id                  BIGINT PRIMARY KEY AUTO_INCREMENT,
    station_code        VARCHAR(16) NOT NULL UNIQUE,    -- 如 FAST_01
    station_type        VARCHAR(8) NOT NULL,            -- FAST / SLOW
    power_kw            DECIMAL(6,2) NOT NULL,
    status              VARCHAR(20) NOT NULL DEFAULT 'IDLE',
    available_time      DATETIME,
    current_request_id  BIGINT,
    last_fault_time     DATETIME,
    created_at          DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at          DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);
```

#### 3.1.4 充电会话表 `charging_session`

```sql
CREATE TABLE charging_session (
    id                  BIGINT PRIMARY KEY AUTO_INCREMENT,
    request_id          BIGINT NOT NULL,
    user_id             BIGINT NOT NULL,
    station_id          BIGINT NOT NULL,
    start_time          DATETIME NOT NULL,
    end_time            DATETIME,
```

```sql
CREATE TABLE billing_record (

    billing_mode    VARCHAR(16) NOT NULL,    -- ENERGY / TIME / MIXED
    energy_fee      DECIMAL(10,2) NOT NULL DEFAULT 0,
    time_fee        DECIMAL(10,2) NOT NULL DEFAULT 0,
    occupancy_fee   DECIMAL(10,2) NOT NULL DEFAULT 0,
    billing_energy  DECIMAL(10,2) NOT NULL DEFAULT 0,
    total_fee       DECIMAL(10,2) NOT NULL DEFAULT 0,
    payment_status  VARCHAR(16) NOT NULL DEFAULT 'UNPAID',  -- UNPAID / PAID / FAILED
    created_at      DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (request_id) REFERENCES charge_request(id),
    FOREIGN KEY (session_id) REFERENCES charging_session(id)
);
```

#### 3.1.7 通知表 `notification`

```sql
CREATE TABLE notification (
```

### 3.2 内存数据结构

为提高调度效率，以下数据结构在运行期维护于内存中，与数据库保持同步。

#### 3.2.1 等待池优先队列

```
fast_pool_heap: 最大堆，按 priority_score 排序
slow_pool_heap: 最大堆，按 priority_score 排序
```

- 堆顶为当前最高优先级的请求
- 每次重调度时重建或更新

#### 3.2.2 充电桩最早可用堆

```
fast_station_heap: 最小堆，按 available_time 排序
slow_station_heap: 最小堆，按 available_time 排序
```

- 堆顶为最早空闲的同类型充电桩
- 桩状态变更时更新堆

#### 3.2.3 事件队列

```
event_queue: FIFO 队列
```

- 接收各类业务事件
- 调度引擎从队列消费事件并执行重调度

### 5.2 调度流程伪代码

```
function reschedule(event):
    // 1. 根据事件类型更新状态
    switch event.type:
        case NEW_REQUEST:
            add_to_pool(event.request)
        case CANCEL:
            remove_from_pool(event.request_id)
            release_station_if_reserved(event.request_id)
        case CONFIRM:
            lock_station_assignment(event.request_id)
        case NO_SHOW:
            release_station(event.station_id)
            handle_no_show_requeue(event.request_id)
        case CHARGE_START:
            update_station_status(event.station_id, CHARGING)
        case CHARGE_END:
            update_station_status(event.station_id, WAITING_TO_LEAVE)
            update_station_available_time(event.station_id)
        case CHARGE_INTERRUPT:
            release_station(event.station_id)
            settle_session(event.request_id)
        case EARLY_COMPLETE:
            update_station_available_time(event.station_id)  // 提前可用
        case LEAVE_TIMEOUT:
            extend_station_available_time(event.station_id)
        case STATION_FAULT:
            mark_station_fault(event.station_id)
            requeue_affected_user(event.station_id)
        case STATION_RECOVER:
            mark_station_idle(event.station_id)

    // 2. 重算优先级
    for pool in [FAST_POOL, SLOW_POOL]:
        update_priorities(pool)

    // 3. 重新预测排程
    for pool_type in [FAST, SLOW]:
        run_prediction(pool_type)

    // 4. 检查叫号
    for pool_type in [FAST, SLOW]:
        check_and_call(pool_type)

function run_prediction(pool_type):
    requests = get_all_requests_sorted_by_priority(pool_type)
    stations = get_available_stations(pool_type)
    // 滚动模拟分配
    station_timeline = {s: available_time(s) for s in stations}
    for request in requests:
        best_station = station with min(station_timeline[s])
        request.estimated_wait = max(0, station_timeline[best_station] - now)
        request.estimated_start = max(now, station_timeline[best_station])
        request.estimated_finish = request.estimated_start + service_time(request, best_station)
        station_timeline[best_station] = request.estimated_finish + leave_buffer

function check_and_call(pool_type):
    for station in get_stations(pool_type):
        if station.status == IDLE or time_until_available(station) < T_call:
            if station not already reserved:
                best_request = peek_top(pool_type)
                if best_request exists:
                    call_user(best_request, station)
```

### 5.3 定时任务

| 任务 | 周期 | 职责 |
|------|------|------|
| 过号检测 | 每 30 秒 | 检查 `CALLED` 状态超过 `confirm_timeout` 的请求 |
| 到场超时检测 | 每 30 秒 | 检查 `CONFIRMED` 状态超过 `arrival_timeout` 的请求 |

## 九、关键设计约束与说明

### 9.1 设计约束

1. **中央等待池**：不采用固定充电桩等待队列，所有请求按类型进入统一池
2. **动态分配**：桩只维护自身状态，不绑定固定用户队列
3. **事件驱动**：调度器仅在关键事件发生时触发，非周期性轮询
4. **联合验收兼容**：所有对外接口严格遵循统一规范
5. **模拟环境**：充电桩和支付为模拟实现
6. **`battery_limit_energy`**：由系统维护，模拟车辆实际可充入上限

### 9.2 性能考虑

1. 等待池使用堆结构，优先级查询和更新为 O(log n)
2. 充电桩可用堆使用最小堆，最早可用桩查询为 O(1)
3. 批量模拟采用事件回放方式，避免实时等待
4. 数据库查询使用索引优化

### 9.3 可扩展性

1. 调度参数全部可配置，支持运行时调整
2. 计费规则可配置，支持多种计费模式
3. 充电桩数量和类型可动态管理
4. 评测指标可扩展
