## 一、文档定位

当前 `develop` 分支中的调度模块实现，不是文档意义上的最终完整版算法，但已经完成了以下目标：

1. 单请求预测不再依赖接口层临时公式
2. 关键事件后的重调度已有统一入口
3. 批量模拟已有可运行的最小执行内核
4. 调度入口和最小返回字段已经具备稳定边界

因此，当前阶段的正确协作方式不是“等待算法最终版后再接入”，而是：

**成员B先接入当前稳定入口，成员A后续继续增强算法内核，保证入口和核心输出字段保持兼容。**

---

## 二、当前算法内核的真实状态

### 2.1 已完成的部分

当前 `develop` 中的调度模块已经具备以下能力：

1. 单请求预测
2. 关键事件后的统一重调度
3. 批量模拟最小执行内核
4. 评分、选桩、叫号建议、状态推进的服务层收口骨架

### 2.2 当前仍为简化版的部分

当前实现仍然是“课程项目当前阶段的最小可用版本”，主要简化包括：

1. 优先级评分采用可扩展的简化公式，而非最终完整版策略
2. 选桩策略采用“预测完成时间最优 / 最早可用桩优先”的简化贪心方式
3. 状态机推进仍保留联调阶段的简化语义
4. 批量模拟仍以最小可用回放为主，而非完整高精度仿真

说明：

- 简化版不等于占位版
- 简化版已经可供后端主流程和批量模拟接入
- 后续增强应发生在调度模块内部，而不是回退到接口层重写临时预测

## 三、当前稳定入口

以下函数应视为当前阶段的稳定调度边界：

| 函数 | 作用 | 当前是否可供成员B接入 |
|------|------|------------------------|
| `predict_request_by_id` | 读取指定请求的最新预测结果 | 是 |
| `reschedule_waiting_requests` | 对当前等待请求执行统一重调度 | 是 |
| `trigger_reschedule_for_event` | 事件驱动的重调度包装入口 | 是 |
| `simulate_batch_case` | 批量模拟执行入口 | 是 |
| `advance_request_state_for_status_view` | 状态查询时使用的最小状态推进器 | 是 |

这些函数当前已经能够支撑：

1. 创建请求后的真实预测字段返回
2. 取消/中断/挪车等事件后的统一重调度
3. 批量模拟的统一输入和统一输出
4. 状态查询所需的最小状态推进

## 四、扩展后接口一致性的要求

### 4.1 必须保持不变的内容

成员A后续在完善算法时，以下内容必须尽量保持兼容：

1. 调度入口函数名
2. 调度入口的调用时机
3. 单请求预测的最小输出字段
4. 批量模拟的输入结构
5. 批量模拟的 `summary + results` 输出结构
6. `request_time` 作为业务时间的语义

也就是说，后续算法增强应遵循：

**内部实现可以升级，外部协作边界不应随意变化。**

### 4.2 允许继续变化的内容

后续仍可继续演进的内容包括：

1. 优先级公式本身
2. 评分参数和权重
3. 选桩策略
4. 全量重算或增量重算的内部优化
5. 批量模拟内部事件回放细节
6. 更复杂异常场景的内部处理细节

只要上述变化不破坏第三章列出的稳定入口，成员B不应被迫返工接口层。

## 五、成员B当前应如何接入

### 5.1 正确做法

成员B在当前 `develop` 基线上继续开发时，应当：

1. 在请求创建后调用当前预测入口
2. 在关键事件后调用当前重调度入口
3. 在批量模拟接口中调用当前批量模拟入口
4. 将自己的工作集中在：
   - 业务流程编排
   - 路由和参数校验
   - 数据持久化
   - 账单与详单输出
   - 场景配置管理
   - 管理端配套接口

### 5.2 错误做法

以下做法应明确禁止：

1. 因为算法还不是最终版，就在接口层重新写临时预测公式
2. 因为批量模拟还可继续增强，就改回纯占位结果
3. 因为状态机仍有简化，就绕开统一重调度入口
4. 以“等成员A拍板”为由回退当前 `develop` 已有能力

## 六、为什么成员B现在接入不会返工

当前内核已经做过一轮“可扩展性收口”，核心思路是：

1. 将评分逻辑与选桩逻辑从主流程中拆开
2. 将状态推进逻辑尽量从路由层收回服务层
3. 保留统一入口函数，不要求成员B直接依赖内部数据结构
4. 允许成员A后续只修改调度模块内部算法，而不改动成员B对外调用方式

因此，后续若成员A完善：

- 优先级公式
- 站桩选择策略
- 故障补偿策略
- 批量模拟回放精度

理论上应主要修改调度模块内部，而不要求成员B改动：

1. 路由路径
2. 参数字段
3. 调度入口调用方式
4. 结果字段解析逻辑

---

## 七、仍然存在的边界风险

虽然当前已经具备较好的扩展空间，但仍需说明：

1. 当前部分状态机语义仍带有联调阶段的简化处理
2. 若未来把叫号、预留、确认、开始充电的完整业务链做得更细，可能需要少量业务流程代码协同调整
3. 这种协同调整应理解为“业务规则细化”，而不是“成员B返工接入接口”

也就是说：

**未来允许少量业务流程协同调整，但不应出现因为算法升级而整体返工接口层的情况。**

---

## 八、协作结论

当前阶段可以明确以下结论：

1. 当前算法内核是简化版，但不是占位版
2. 当前算法内核已经具备成员B先行接入的条件
3. 成员A后续继续完善算法时，应以保持调度入口和最小输出兼容为原则
4. 成员B当前不应等待算法最终版后再开发，而应在 `develop` 基线上先接入
5. 后续若成员B因算法仍未最终完成而回退为接口层临时预测，视为破坏分工边界

最终原则可概括为：

**先接入当前稳定内核，后增强内部算法；保持接口一致性，避免成员B返工，防止分工耦合。**
