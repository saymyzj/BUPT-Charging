## 一、项目目标与设计原则

### 1.1 项目目标

本系统面向校区智能充电场景，目标是在模拟环境下完成充电请求受理、动态调度、排队管理、计费结算和管理监控，并尽量实现以下优化目标：

- 缩短用户平均等待时间
- 缩短用户平均完成时间
- 提高充电桩利用率
- 在异常场景下保持调度稳定性和公平性

### 1.2 对题目要求的理解

题目提出“使得充电和等待时间达到最短的效果”。结合课程项目可实现性，我们将其具体化为：

- 主目标：最小化用户总完成时间  
  总完成时间 = 等待时间 + 实际充电时间
- 次目标：在保证基本公平性的前提下，尽量降低平均等待时间并提高系统吞吐量

说明：

- 用户提交的核心输入为：充电类型、申请充电量
- 用户上报的申请充电量视为预测输入，而非最终精确结算值
- 充电时间可以由“申请充电量 / 充电桩功率”估算得出
- 调度算法主要优化等待时间和资源分配，不直接改变用户所需电量
- 最终计费和会话结算以实际充电量为准

### 1.3 设计原则

基于前期分析，本系统不采用“用户一提交请求就固定绑定某个充电桩队列”的方案，而采用：

`中央等待池 + 动态调度 + 临近确认 + 事件驱动重排`

这样设计的原因是：

- 固定绑定某个桩队列，难以处理取消、过号、故障、充电中断等动态事件
- 统一等待池更适合重新计算预计等待时间
- 动态分配最终桩号，更符合真实调度系统的运行方式

### 1.4 最终验收对项目设计的影响

根据老师最新确认，项目最终采用系统验收方式，不再进行多组联合验收。

因此，项目设计需要满足以下要求：

- 充电桩数量不是固定值，而是验收时可灵活设置的超参数
- 慢充桩数量、快充桩数量、等待区车位容量都应纳入系统配置
- 测试用例可能先以自然语言给出，再映射为系统输入
- “充电桩排队队列长度”这一描述存在歧义，系统需兼容两种解释：
  1. 每个充电桩的统一排队容量
  2. 每个充电桩各自的初始状态与排队人数快照
- 最终验收更关注系统本身是否能正确建模、调度、计费和输出结果，而不是跨组通信协作

设计上的直接影响是：

- 文档和实现需要把“场景参数配置”视为核心能力，而不是固定样例
- 调度模型内部仍可采用中央等待池，但对外输入必须兼容验收方给出的不同队列描述
- 增量二的重点应转为验收场景适配、参数化配置、异常修正和展示优化

## 二、前提条件一：需求框架确认

### 2.1 角色定义

| 角色 | 描述 | 核心需求 |
|------|------|----------|
| 普通用户 | 校园内发起充电请求的用户 | 提交请求、查看预计等待时长、接收叫号、查看详单与账单 |
| 管理员 | 负责系统运行和资源管理的人员 | 监控桩状态、处理异常、配置参数、查看统计、管理用户与设备 |



### 2.2 用户提供的数据

本系统以尽量简化用户输入为原则。用户发起申请时，至少提供以下数据：

| 字段 | 含义 |
|------|------|
| charge_mode | 充电类型，取值为快充或慢充 |
| request_energy | 申请充电量，单位可设为 kWh，作为调度预测输入 |

系统据此计算：

- 预计充电时长
- 预计等待时长
- 预计开始时间
- 预计完成时间

说明：

- 用户未必能准确知道最终需要充入多少电量，因此 `request_energy` 只作为调度阶段的估算依据
- 系统在充电过程中实时记录 `actual_energy`
- 若车辆提前充满，则会话提前结束，充电桩提前释放，并触发重调度
- 最终账单按 `actual_energy` 和实际充电时长计算

### 2.4 核心业务流程

系统核心业务主线如下：

1. 用户提交充电请求
2. 用户进入对应类型的中央等待池
3. 调度器根据当前桩状态和等待池状态计算预计等待时间
4. 当某个充电桩即将空闲时，系统选择合适用户进行临近确认
5. 用户确认后，系统最终分配充电桩号
6. 用户开始充电，系统记录会话并实时计费
7. 充电结束后进入待挪车或直接释放状态
8. 系统生成详单与账单
9. 若实际充电量提前达到车辆上限，系统提前结束本次会话
10. 若中途发生取消、过号、中断、故障等事件，触发重调度

补充说明：

- 在系统启动或批量模拟开始前，需要先加载本次验收场景参数
- 场景参数至少包括：`fast_station_count`、`slow_station_count`、`waiting_area_capacity`
- `waiting_area_capacity` 表示系统全局等候区容量，由快充等待池和慢充等待池共享
- 若验收样例直接给出每个桩的排队状态，则系统先把该快照转换为内部初始状态，再开始调度

### 2.5 系统边界

- 充电桩采用模拟设备，不接入真实硬件
- 用户是否真正到场、是否挪车等行为由系统状态模拟
- 支付采用模拟支付
- 本项目以课程实验环境为主，不追求生产级吞吐和高可用

## 四、前提条件三：系统总体结构设计确认

### 4.1 模块划分

| 模块 | 主要职责 |
|------|----------|
| 用户服务模块 | 注册、登录、身份识别 |
| 请求受理模块 | 接收充电请求，生成调度任务 |
| 等待池管理模块 | 维护快充池和慢充池中的请求 |
| 调度引擎模块 | 计算优先级、预计等待时间和最终分配 |
| 充电桩状态模块 | 维护充电桩当前状态和最早可用时间 |
| 充电会话模块 | 维护充电中、完成、挪车等状态 |
| 计费与账单模块 | 根据会话记录生成费用与账单 |
| 通知模块 | 叫号、提醒、异常通知 |
| 管理后台模块 | 提供监控、配置、统计和审计能力 |

### 4.2 总体调度架构

本系统采用“两类等待池 + 多个充电桩状态机 + 一个中央调度器”的总体架构：

- 快充请求进入快充等待池
- 慢充请求进入慢充等待池
- 每个充电桩只维护自己的实时状态，不维护固定用户队列
- 调度器在关键事件发生时统一重算预计等待时间与分配结果

为了兼容最终验收输入，还需补充一层“场景适配”：

- 若验收给出统一队列长度，则将其解释为每桩排队容量上限
- 若验收给出桩级状态快照，则将其解释为每个桩的初始占用和排队情况
- 以上两种输入最终都转换为内部的等待池状态和桩状态，再进入统一调度流程

## 五、核心调度算法说明

### 5.1 算法总体思路

本系统不采用“用户一提交就固定绑定某个充电桩队列”的做法，而是采用：

`中央等待池 + 动态调度 + 临近确认 + 事件驱动重排`

具体来说：

- 快充用户进入快充等待池，慢充用户进入慢充等待池
- 系统先计算预计等待时间，不立即锁定具体桩号
- 当某个同类型充电桩即将可用时，再根据当前状态选择最合适的用户
- 若发生取消、过号、提前充满、故障等事件，立即触发重调度

这样可以在保持公平性的前提下，更灵活地缩短单用户本次总时长，并兼顾整个系统的总完成时长。

### 5.2 核心公式

1. 预计服务时长  
`service_time(u, s) = remaining_energy(u) / power(s)`

若同类型桩功率相同，也可简化为：

`service_time(u) = remaining_energy(u) / power(mode(u))`

2. 实际充电量  
`actual_charge_energy(u) = min(request_energy(u), battery_limit(u))`

3. 实际服务时长  
`actual_service_time(u, s) = setup_time + actual_charge_energy(u) / power(s)`

4. 充电桩最早可用时间  
`available_time(s) = charge_end_time(s) + leave_buffer`

5. 预测等待时间  
`pred_wait(u, s) = max(0, available_time(s) - now)`

6. 预测完成时间  
`pred_finish(u, s) = pred_wait(u, s) + service_time(u, s)`

7. 动态优先级  
`priority(u) = a * wait_time(u) + b * retry_bonus(u) + c * fault_bonus(u) - d * service_time(u) - e * no_show_penalty(u)`

说明：

- `request_energy` 用于预测，`actual_charge_energy` 用于最终结算
- 若用户提前充满，则会话提前结束，充电桩提前释放，并更新后续用户预测结果
- `a、b、c、d、e` 为可配置权重，用于平衡效率与公平性
- 普通目标优先关注单用户的 `wait_time + actual_service_time`
- 扩展目标关注所有用户总完成时间之和，可通过统计指标验证

### 5.3 算法流程

1. 用户提交 `charge_mode` 和 `request_energy`
2. 系统将请求加入对应类型的中央等待池
3. 调度器读取同类型各充电桩的 `available_time`
4. 系统根据优先级公式计算每个请求的 `priority`
5. 按优先级和最早可用桩进行滚动预测，生成预计等待时间
6. 当某个桩接近空闲时，系统向当前最合适的用户发起叫号确认
7. 用户确认后，系统分配最终桩号并开始充电
8. 充电过程中若提前充满、用户取消、中断、过号或桩故障，则触发重调度
9. 充电结束后生成详单与账单

### 5.4 关键处理说明

- 用户申报量与实际量不一致时，系统按实际充电量结算
- 用户过号时，可重新入池并附加轻度惩罚
- 用户超时未挪车时，充电桩保持不可用，并可计入占位费
- 桩故障时，相关用户重新入池并获得一定补偿优先级

详细的异常规则、状态机设计和数据结构设计见 `调度算法.md`。
