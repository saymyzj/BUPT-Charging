# 前端兜底功能的后端接口需求

## 背景

当前前端已在不修改后端代码的前提下，对部分功能做了临时兜底。但这些兜底方案不能保证数据完整性，也无法覆盖跨浏览器、跨设备、清除本地缓存等场景。

本文档只记录目前仍需要后端补充的能力。

## 1. 当前用户历史账单列表接口

### 现状

账户中心需要展示“当前用户自己的所有账单”。

目前后端已有单个详单查询接口：

```http
GET /api/request/detail/{request_id}
```

但普通用户侧没有“查询自己所有历史账单”的列表接口。

因此前端目前只能临时把用户提交过的 `request_id` 存在浏览器本地，然后逐个调用单个详单接口查询。

### 前端兜底方案的限制

当前兜底方案存在以下问题：

- 只能展示当前浏览器记住过的请求。
- 用户换浏览器或换设备后，历史账单无法恢复。
- 用户清除浏览器缓存后，本地 `request_id` 记录会丢失。
- 用户在其他设备提交过的请求，当前设备无法知道对应 `request_id`。
- 请求未完成、后端未生成详单时，前端无法展示账单。

所以该方案不能作为最终的“所有账单”能力。

### 需要后端提供

建议新增普通用户账单列表接口：

```http
GET /api/request/details
```

或：

```http
GET /api/auth/profile/bills
```

### 权限要求

- 只能返回当前登录用户自己的账单。
- 当前用户身份应由登录态或 token 解析。
- 前端不应传 `user_id` 来决定查询谁的账单。
- 管理员查看任意用户账单应继续使用管理员接口，和普通用户接口分开。

### 建议响应

```json
{
  "code": 0,
  "data": [
    {
      "detail_id": "DETAIL0001",
      "request_id": "REQ0001",
      "user_id": "U001",
      "station_code": "FAST_01",
      "charge_mode": "FAST",
      "actual_energy": 12.5,
      "charging_duration_minutes": 25,
      "start_time": "2026-05-04T23:41:00",
      "stop_time": "2026-05-04T23:45:00",
      "charging_fee": 8.75,
      "service_fee": 10.0,
      "total_fee": 18.75,
      "termination_status": "COMPLETED_EARLY",
      "detail_generated_at": "2026-05-04T23:45:00"
    }
  ]
}
```

### 排序与分页

建议默认按 `detail_generated_at` 倒序返回，最新账单在前。

如果后续账单数量较多，可以支持分页：

```http
GET /api/request/details?page=1&page_size=20
```

### 前端接入方式

后端提供该接口后，前端账户中心应改为：

- 优先调用新的历史账单列表接口。
- 不再依赖浏览器本地保存的 `request_id` 作为主数据源。
- 本地 `request_id` 记录仅作为兼容兜底。
- 单个账单详情页继续使用：

```http
GET /api/request/detail/{request_id}
```

## 2. 当前活跃请求恢复接口

### 现状

前端目前依赖本地保存的 `request_id` 来恢复当前请求状态。

如果用户退出登录、换设备、清除缓存，或者之前请求是在其他浏览器提交的，前端可能只知道“当前用户已有活跃请求”，但不知道具体的 `request_id`，无法直接展示请求详情。

### 前端兜底方案的限制

当前兜底只能提示用户：

- 当前用户已有活跃请求。
- 但本地缺少请求编号，无法直接展示详情。
- 需要进入当前请求页查看提示，或由管理员结束该请求后重试。

这不是完整体验。

### 需要后端提供

建议新增当前用户活跃请求查询接口：

```http
GET /api/request/active
```

### 权限要求

- 只能返回当前登录用户自己的活跃请求。
- 用户身份由登录态或 token 解析。
- 如果没有活跃请求，返回明确的空状态。

### 建议响应

有活跃请求：

```json
{
  "code": 0,
  "data": {
    "request_id": "REQ0001",
    "queue_number": "F1",
    "request_status": "CHARGING",
    "charge_mode": "FAST",
    "request_energy": 30,
    "front_waiting_count": 0,
    "estimated_wait_seconds": 0,
    "estimated_finish_time": "2026-05-05T00:15:00",
    "station_code": "FAST_01"
  }
}
```

无活跃请求：

```json
{
  "code": 0,
  "data": null
}
```

### 前端接入方式

后端提供该接口后，前端应在以下场景主动调用：

- 用户登录成功后。
- 进入工作台页面时。
- 进入当前请求页面时。
- 提交请求返回“已有活跃请求”错误时。

如果接口返回活跃请求，前端保存 `request_id` 并展示真实请求状态。

## 3. 用户账单与请求历史的关系

历史账单列表接口只负责已生成详单的记录。

如果还需要展示未完成请求、已取消请求、等待中请求等完整历史，则建议另行提供请求历史接口：

```http
GET /api/request/history
```

该接口可返回所有请求，不限于已生成账单的请求。

账户中心的“我的账单”优先使用账单列表接口；当前请求页使用活跃请求接口；如后续需要“请求历史”，再接入请求历史接口。

## 4. 账单支付与支付状态接口

### 现状

当前正式接口口径中，详单生成后即作为最终账单展示，未单独实现支付流程。

也就是说，用户现在点击“查看详单”后，前端可以展示费用明细和支付按钮，但没有真实的后端支付状态，也没有可以调用的支付接口。

当前前端已增加“立即支付”按钮，并使用浏览器本地存储做临时模拟支付状态。

如果业务要求用户查看详单后必须完成支付或确认缴费，那么该能力需要后端支持。前端只能做按钮和提示，不能真实改变账单支付状态。

### 前端兜底方案的限制

如果只在前端本地模拟支付，会存在以下问题：

- 刷新缓存、换浏览器或换设备后，支付状态会丢失。
- 管理端无法可信统计哪些账单已支付。
- 后端报表无法区分待支付、已支付、支付失败等状态。
- 用户重复点击或并发操作时，前端无法保证账单只支付一次。
- 无法对支付金额、账单状态、支付时间进行后端审计。

因此支付状态不能只靠前端本地保存。

### 需要后端提供

建议新增账单支付接口：

```http
POST /api/request/pay
```

请求体建议：

```json
{
  "request_id": "REQ0001",
  "pay_time": "2026-05-05T00:20:00"
}
```

如果后端以详单编号作为支付对象，也可以使用：

```json
{
  "detail_id": "DETAIL0001",
  "pay_time": "2026-05-05T00:20:00"
}
```

### 支付状态字段

账单详情和账单列表接口建议增加支付状态字段：

```json
{
  "payment_status": "UNPAID",
  "paid_at": null
}
```

支付成功后：

```json
{
  "payment_status": "PAID",
  "paid_at": "2026-05-05T00:20:00"
}
```

建议状态枚举：

| 状态 | 含义 |
| --- | --- |
| `UNPAID` | 待支付 |
| `PAID` | 已支付 |
| `PAY_FAILED` | 支付失败 |

如当前项目只做模拟支付，可以暂时只实现 `UNPAID` 和 `PAID`。

### 权限要求

- 普通用户只能支付自己的账单。
- 已支付账单不能重复支付。
- 未生成详单的请求不能支付。
- 已取消且未生成详单的请求不能支付。
- 支付金额应以后端账单金额为准，不能以前端传入金额为准。

### 建议响应

支付成功：

```json
{
  "code": 0,
  "message": "支付成功",
  "data": {
    "request_id": "REQ0001",
    "detail_id": "DETAIL0001",
    "payment_status": "PAID",
    "paid_at": "2026-05-05T00:20:00",
    "total_fee": 18.75
  }
}
```

重复支付：

```json
{
  "code": 1009,
  "message": "账单已支付"
}
```

### 前端接入方式

后端提供支付能力后，前端应调整为：

- 账单详情页展示支付状态。
- `UNPAID` 状态展示“立即支付”按钮。
- 点击支付后调用 `POST /api/request/pay`。
- 支付成功后刷新当前详单和账户中心账单列表。
- 历史账单列表展示“待支付 / 已支付”状态。
